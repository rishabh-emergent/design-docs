CAP Margin Capture via Billing Decoupling

1. TL;DR (for the impatient)

Right now, our Cache Anchor Point (CAP) system saves Anthropic money by keeping the prompt cache warm. But the way we bill users today means those savings flow through to the user — meanwhile, CAP itself costs us money in background "ping" requests. Net result: Emergent loses margin on every CAP-active user.

This work makes one targeted change: when CAP is the reason the cache survived, charge the user as if the cache had naturally died, but pay Anthropic the actually-lower bill. The user's bill is unchanged from a CAP-free world — they pay exactly what they would have paid if CAP didn't exist. The savings become Emergent margin.

What this is NOT

  • Not a price increase for users. Users pay the same as before CAP existed.
  • Not a new line item on bills. No SKU change. No UI change.
  • Not a CAP rework. CAP behavior, intervals, models — all unchanged.

2. Zoom-out: how prompt caching works (today, without CAP)

Anthropic has a feature called prompt caching. The idea: if you send the same long prompt prefix many times in a short window, Anthropic caches it on their side and charges a lower rate when you re-send it.

Three price tiers

TierRate (relative)When you pay it
Uncached input1.00×Tokens that are not covered by a cache breakpoint (rare in our system — we always set breakpoints).
Cache write1.25×The first time a cached prefix is sent, OR after the cache has expired and you send the same prefix again.
Cache read0.10×When the prefix is already in Anthropic's cache and you send the same prefix again within the TTL window.

So cache reads are 12.5× cheaper than cache writes. That's the entire reason caching matters.

The 5-minute TTL

Anthropic's prompt cache has a Time-To-Live of 5 minutes. If you wait more than 5 minutes between requests with the same prefix, the cache is gone — your next request becomes a cache write again (1.25×).

Calls 1, 2, 3 land inside the 5-minute window so calls 2 and 3 are cheap cache reads. Call 4 arrives after the TTL — cache is gone, Anthropic charges full cache-write again.

3. Zoom-out: what CAP does (today)

Users are humans. Humans pause to read, to think, to take a coffee break. When a user pauses for 10 minutes between agent turns, the 5-minute TTL expires and the next agent turn pays 1.25× on the entire prefix again.

CAP (Cache Anchor Point) is our trick to prevent that. Every ~4 minutes 50 seconds, CAP fires a small background LLM call (a "ping") with the same prompt prefix. That ping resets the cache TTL on Anthropic's side. When the user comes back 10 minutes later, the prefix is still cached and the user's call pays 0.10× instead of 1.25×.

Without CAP, a 10-minute user pause kills the cache. With CAP, two pings ~4m50s apart keep the cache continuously alive.

5. The decoupling idea (the fix)

We change one thing: when CAP is the reason the cache survived, we charge the user as if the cache had died naturally.

6. Vocabulary (the y / z / x triad)

From the PRD and from CONTEXT.md. These three names show up in every diagram, every audit row, every report. Learn them once and the rest of the doc is easier to read.

y — actual-cost
What Anthropic actually charged us for this user call. Computed by the xllm pricing calculator from the call's real Usage (with whatever cache reads and writes it actually had).
z — as-if-cold-cache
What this call would have cost if the cache had naturally invalidated before it. Computed by taking the same Usage, swapping all cache_read_tokens into cache_creation_tokens, and re-running the pricing calculator. z ≥ y, always.
x — ping-cost
The full cost of the most-recent CAP ping that returned cache_hit=true between the previous user call and this one. Computed per-ping inside the CAP activity.
cap-decoupled call
A user LLM call where billing is decoupled — user pays z, Emergent pays Anthropic y, Emergent absorbs x, and the margin captured is z − y − x.
natural-cache-ttl
Anthropic's prompt cache TTL: 5 minutes. Lives in code as cap.CacheTTL in core/cap/config.go.

margin = z − y − x

7. The attribution rule (when do we decouple?)

We do not decouple every CAP-active call. We have to be careful: if we charge the user at z on a call where the cache would have lived naturally anyway, we'd be overcharging the user.

So we use a strict rule. A call is cap-decoupled if and only if both of these are true:

  1. Elapsed since the user's last LLM call > 5 minutes. If they came back within 5 minutes, the cache would still be alive naturally — no decoupling.
  2. At least one CAP ping fired in that gap returned cache_hit=true. "In that gap" is the half-open interval (last_non_ping_llm_call_at, this_call's_llm_return_time] — bounded on both ends so pings that fire after this LLM responded (in-loop refresh during a long tool execution) don't get misattributed to the call that preceded them.

Why so strict? Because false positives are visible to users (overcharging), but false negatives are invisible (we just don't capture margin we could have). When in doubt, bill at y.

9. Data flow (sequence diagram)

Two flows woven together: the CAP ping flow (writes cap_ping rows to BigTable's ping_data column family) and the user-call flow (loads pings once at activityLoadThread, resolves attribution per iteration in activityPostHook, swaps cost_usd on the existing PostHook payload). The per-call attribution audit lives entirely inside cortex — structured Loki logs ("cap attribution resolved") plus OTel metrics (cap.attribution.*) — nothing CAP-specific crosses the cortex/emergent boundary other than the swapped cost_usd value.

Sequence diagram for Scenario B. callTime is the LLM-return time captured in workflow code immediately after ActivityCallLLM completes (not the post-hook wall time). The pre-loaded ping list is filtered in-memory: candidate must be source=post_exit, cache_hit=true, fired in (prevAt, callTime], and within CacheTTL of callTime. Emergent's billing pipeline is unchanged — it just reads the swapped cost_usd off the existing PostHook payload and bills against credits as usual. No new field crosses the cortex/emergent boundary.

10. Storage model (BigTable column family + cortex telemetry)

Post-launch update (PR cortex#1483, 2026-05-23)

v2 originally read the eligibility boundary and the cold-cache baseline from messages on every workflow setup. That saturated cortex-db CPU at 70–90% within hours. Both reads now live in a Redis hash session:{sessionID}:meta — one HMGET per setup, in parallel with the thread load. See section 11 for the full Redis-runtime-meta design and visualization. The BigTable ping_data column family described below is unchanged.

The design adds no new Postgres tables or columns anywhere. Two cortex-local storage locations carry the new state:

  1. Cortex BigTable (existing cortex-sessions table, dedicated ping_data column family) holds each fired CAP ping as one row. Row key cap_ping#<sessionID>#<firedAtMillis_13d>#<requestID> — sorted lexicographically chronological, range-scanned (not full-prefix) by session. The column family has its own GC rule (MaxAge = 7d) so ping retention is bounded independent of session/message data in the same table.
  2. Cortex telemetry pipeline — per-call attribution lands in two places: structured Loki logs ("cap attribution resolved" at elite/activities.go:780) and OTel metrics (cap.attribution.total, cap.attribution.margin_usd, cap.attribution.billable_usd; introduced in cortex#1465) flowing through prometheus-prod. No emergent-side schema change.
Two cortex-local storage locations. BigTable ping_data column family: one row per fired ping under cap_ping#<session>#<ms>#<reqID>, GC-bounded to 7 days. Cortex telemetry: structured Loki logs + Prometheus histograms/counters carry the per-call audit. The PostHook payload to emergent stays bytewise compatible with pre-CAP — only cost_usd and accumulated_cost shift to z when decoupled.

Why this storage shape

  • BigTable ping_data column family: writes are append-only key-value puts. Row key is sortable so a single GetByRange(cap_ping#<session>#<startMs>#, cap_ping#<session>$) returns only the rows after the previous user message in chronological order. The end-key uses $ (0x24 > # 0x23) to bound the session's range exclusively. No full-prefix scan, no JOINs, no schema migration, no per-call DB round trip during the hot path.
  • Loaded once per workflow at setup, in parallel with the global-cache baseline: cap.Service.LoadContext issues the BigTable range scan and the head-message baseline SQL query as two goroutines (each wrapped in ego.SafeExec for panic recovery), then caches both on setup.Cap.Pings / setup.Cap.GlobalCacheBaseline. Per-iteration PostHook reads the in-memory cache.
  • Audit lives in cortex telemetry: structured Loki logs carry full per-call detail (job_id, session_id, decoupled, reason, y, z, x, billable, baseline); OTel metrics emit aggregates suitable for Grafana panels. No emergent-side persistence — joining ping_request_id to trajectories is no longer needed.
  • Dedicated 7-day TTL: the ping_data column family has its own GC rule (MaxAge = 7d) so ping retention is decoupled from session/message retention on the same table. Reporting windows past 7 days require BigQuery export, not the live store.
  • Failsafe to user: BigTable read failure → empty ping list → no_qualifying_ping for the session → bills y. Postgres prev-msg-time read failure → attribution_unavailable → bills y. Pricing calculator failure (z = 0) → swap is gated on BillableCostUSD > 0 so the user is never billed nothing. No path bills the user more than the no-CAP world.

11. Redis runtime meta (PR cortex#1483)

v2 shipped with two Postgres reads on every workflow setup — GetPreviousAssistantMessageTime (the CAP eligibility boundary, MAX(created_at) over messages WHERE role='assistant') and GetHeadMessageCacheReadTokens (the cold-cache baseline, a JSONB scan on the session head message). Together they saturated cortex-db CPU at 70–90 % within hours. PR cortex#1482 commented out the call site as an emergency hot-fix; CAP attribution went dark until #1483 landed two days later.

This section documents the replacement: both reads live in a single Redis hash, written on the hot path, read once at setup.

11.1 What moved and what didn't

Cold-cache baseline (head-msg cache_read_input_tokens) SQL: JSONB scan of the session head message's usage blob, ASC LIMIT 1 — fired per workflow setup Redis hash field system_prompt_cache_read_tokens, written via explicit setter only when the system-prompt context boundary actually changes (init / handoff @ iter 0)
"Any LLM call happened" timestamp (not tracked separately) New Redis hash field last_llm_call_at — advanced by every LLM call, elite OR CAP ping. Used for cache-TTL timing and future non-CAP consumers

11.2 Storage layout

One hash per session in the shared cortex Redis. Pure point-read / point-write — no scans, no sorted sets.

key:    session:{sessionID}:meta
TTL:    365d sliding, refreshed on every write
fields:
  last_llm_call_at                 → int64 unix millis  (any LLM call, elite OR CAP ping)
  last_assistant_msg_at            → int64 unix millis  (elite-driven assistant messages ONLY)
  system_prompt_cache_read_tokens  → int32              (head-baseline cache_read tokens)
  schema_version                   → int8               (currently 1)

Why two timestamp fields, not one

If a single field were updated by both elite and pings, CAP attribution would silently break: every workflow setup would see a ping as "the latest LLM call", the eligibility window (prevAt, callTime] would never contain pings, and decoupling would never fire. The original Postgres query intentionally excluded pings (they don't write to messages); the two-field design preserves that semantic.

11.3 The baseline write trigger (the trap)

system_prompt_cache_read_tokens is the cold-cache cost subtraction used by CalculateColdCacheCost. It must represent "cache_read for the first call that warmed this system-prompt prefix", not "cache_read for the most recent call".

If every LLM call overwrote the field, the baseline would ratchet up as the session warms — and z − y would systematically under-claim margin (the calculator would subtract an inflated baseline from the call's own cache_read, producing a near-zero margin by construction).

The field is therefore written by an explicit setter triggered only when the system-prompt context boundary actually changes:

// core/workflows/elite/request.go
RecordCacheBaseline = (req.IsInitRequest || req.IsHandoff) && iterationNumber == 0
TriggerFlagWhy this is a new system-prompt context
Session creation (Neo → Elite first message)Request.IsInitRequestFirst call ever; system prompt freshly cached.
Summarisation handoff to the infinite agentRequest.IsHandoffNew workflow on a new agent → different system prompt → different cache key.

Everything else — regular resume (same agent, new user message), iter 2..N of any workflow, graceful_finish, subagent calls, CAP refresh pings, in-workflow auto-compact / squashing — leaves the baseline untouched. The prior baseline carries over because the cached system-prompt prefix has not changed.

11.4 Hardcoded fallback (PR cortex#1485)

Pre-existing sessions (born before #1483) have no Redis-set baseline, and brand-new sessions briefly look the same on iter 0 before the first response usage lands. To stop those sessions from degrading to no_qualifying_ping → bill-at-y, EffectiveGlobalCacheBaseline returns a hardcoded floor when the measured value is zero:

// core/cap/global_cache.go
const DefaultGlobalCacheBaseline = 26000  // observed prod range: 21k–28k

func EffectiveGlobalCacheBaseline(measured int) int {
    if measured > 0 { return measured }
    return DefaultGlobalCacheBaseline
}

The eligibility-reason no_global_cache_baseline still exists in the codebase but is now unreachable on the elite path — the fallback closes the gap. It's kept for legibility and tests.

11.5 Data-flow visualization

The hot path makes zero database round-trips after Phase A. Two parallel goroutines at setup populate Redis (point read) and BigTable (range scan); both are wrapped in ego.SafeExec so a Redis miss or BT scan failure downgrades to bill-at-y without crashing the workflow. The CAP ping writes only last_llm_call_at so the elibility boundary remains pinned to the last elite call — without that asymmetry, attribution would silently fail.

12. Code changes (cortex-only)

12.1 core/cap/service.go — single Service abstraction

The entire CAP runtime hangs off one struct exposing three methods. Store is the persistence contract; internal/pgsql.CapStore satisfies it by composing the PG store (for prev-msg lookup and head-message usage) with a dedicated BigTable client targeting the ping_data column family.

// core/cap/service.go
type Store interface {
    LoadCapPings(ctx context.Context, sessionID string, after time.Time) ([]PingRecord, error)
    GetPreviousAssistantMessageTime(ctx context.Context, sessionID, excludeMessageID string) (time.Time, error)
    GetHeadMessageUsage(ctx context.Context, sessionID string) (int, error)
    SavePing(ctx context.Context, sessionID string, record PingRecord) error
}

type Service struct{ store Store }

// NewService wires a real store-backed CAP service.
func NewService(store Store) *Service { return &Service{store: store} }

// NewNoOpService returns a Service whose methods short-circuit to zero values.
// Used in replayer/tests where CAP must be wired but inert; replaces the
// previous nil-guard pattern at every call site.
func NewNoOpService() *Service { return &Service{} }

12.2 cap.Service.LoadContext — parallel BT + baseline at setup

Pings + global-cache baseline load in parallel; each goroutine is wrapped in ego.SafeExec so a panic in either is recovered, logged, and downgraded to a partial result (empty pings or zero baseline → no decoupling this run, never a workflow crash).

// core/cap/service.go (abridged)
func (s *Service) LoadContext(ctx context.Context, sessionID string) LoadContextResult {
    var out LoadContextResult
    if s.store == nil { return out }

    prevAt, err := s.store.GetPreviousAssistantMessageTime(ctx, sessionID, "")
    if err != nil {
        log.Warn(ctx, "cap context: prev-msg-time query failed (non-fatal)",
            log.Str("session_id", sessionID), log.Err(err))
        return out
    }

    var wg sync.WaitGroup
    wg.Add(2)
    go func() {
        defer wg.Done()
        pings, _ := s.store.LoadCapPings(ctx, sessionID, prevAt)  // bounded range scan
        out.Pings = pings
    }()
    go func() {
        defer wg.Done()
        tokens, _ := s.store.GetHeadMessageUsage(ctx, sessionID)
        if IsValidGlobalCacheBaseline(tokens) { out.GlobalCacheBaseline = tokens }
    }()
    wg.Wait()
    return out
}

12.3 internal/pgsql/cap_pings.go — BigTable composite store

The bounded range scan uses $ (0x24) as the end-key sentinel, ensuring lexicographic exclusivity over # (0x23). Only source = post_exit rows are returned for attribution; in-loop pings are still persisted (for telemetry) but filtered out here.

// internal/pgsql/cap_pings.go (abridged)
func capPingRangeAfter(sessionID string, after time.Time) (string, string) {
    var startMs int64
    if !after.IsZero() { startMs = after.UnixMilli() + 1 }
    start := fmt.Sprintf("cap_ping#%s#%013d#", sessionID, startMs)
    end   := fmt.Sprintf("cap_ping#%s$",       sessionID)
    return start, end
}

    // ... unmarshal each row, drop source!=post_exit, return
}

func (s *CapStore) SavePing(ctx context.Context, sessionID string, record cap.PingRecord) error {
    if s.bt == nil { return nil }  // NoOp path — keep callsites uniform

    EligibilityAttributionUnavailable = "attribution_unavailable"
    // Decoupling refused — coldCacheCost would over-claim without a measured baseline.
    EligibilityNoGlobalCacheBaseline  = "no_global_cache_baseline"
)

func (s *Service) ResolveAttribution(ctx context.Context, in ResolveInput) Attribution {
    calc := NewPricingCalculator(ctx)
    actualCost    := CalculateActualCost(calc, in.Model, in.Usage)
    coldCacheCost := CalculateColdCacheCost(calc, in.Model, in.Usage, in.GlobalCacheBaseline)
    failsafe      := Attribution{
        BillableCostUSD: actualCost, ActualCostUSD: actualCost, ColdCacheCostUSD: coldCacheCost,
        EligibilityReason: EligibilityAttributionUnavailable,
    }
    if s.store == nil { return failsafe }

    prevAt, err := s.store.GetPreviousAssistantMessageTime(ctx, in.SessionID, in.CurrentMessageID)
    if err != nil { return failsafe }

    candidate   := PickCandidate(in.Pings, prevAt, in.CallTime)
    pingCostSum := SumPostExitPingCosts(in.Pings, prevAt, in.CallTime)
    decision    := DecideEligibility(in.CallTime, prevAt, candidate)
    // Refuse to decouple without a measured baseline.
    if decision.Reason == EligibilityDecoupled && in.GlobalCacheBaseline == 0 {
        decision.Reason = EligibilityNoGlobalCacheBaseline
    }
    return BuildAttribution(decision, actualCost, coldCacheCost, pingCostSum, candidate)
}

12.5 core/workflows/elite/activities.go — activityPostHook

The CAP swap is gated on BillableCostUSD > 0 — guards against an unobservable z=0 underbill if the pricing calculator returns nothing on the decoupled path while still succeeding on y. The cap.RecordAttribution call (cortex#1465) emits the new OTel histograms alongside the existing structured log.

// core/workflows/elite/activities.go (abridged)
func (w *Workflow) activityPostHook(ctx context.Context, input PostHookActivityInput) (*PostHookActivityResult, error) {
    if !input.CapEnabled {
        resp, err := w.agentSvc.PostHook(ctx, input.Request)
        return &PostHookActivityResult{Response: resp}, err
    }
    attribution := w.capSvc.ResolveAttribution(ctx, cap.ResolveInput{
        SessionID:           input.Iteration.SessionID,
        CurrentMessageID:    input.Iteration.CurrentMessageID,
        Model:               input.Iteration.Model,
        Usage:               input.Iteration.Usage,
        CallTime:            input.Iteration.CallTime,
        GlobalCacheBaseline: input.Iteration.GlobalCacheBaseline,
        Pings:               input.CapPings,
    })
    // Only swap when the chosen path actually priced.
    var costDiff float64
    if attribution.BillableCostUSD > 0 {
        costDiff = attribution.BillableCostUSD - attribution.ActualCostUSD
        input.Request.CostUSD = attribution.BillableCostUSD
        input.Request.AccumulatedCost += costDiff
    }
    log.Info(ctx, "cap attribution resolved", /* y, z, x, billable, reason, baseline */)
    cap.RecordAttribution(ctx, input.Iteration.Model, input.Iteration.Provider, attribution)
    resp, err := w.agentSvc.PostHook(ctx, input.Request)
    return &PostHookActivityResult{Response: resp, CostDiff: costDiff}, err
}

12.7 pkg/agentsvc/hooks.go — no schema change

The PostHook payload to emergent does not carry a CAP-specific field. The audit blob initially planned for traj_payload.cap was dropped in favour of cortex-local telemetry (Loki + Prometheus). Key architectural properties

  • Zero new Temporal activities: attribution logic lives in pure functions on cap.Service, invoked from activityPostHook; ping load lives inside activityLoadThread. Replay determinism is preserved by Temporal's existing activity-result caching.
  • Setup-time load is hot-path-free: BigTable is range-scanned once per workflow, in parallel with the head-message baseline query; both panic-safe via ego.SafeExec. Per-iteration PostHook reads in-memory state only.
  • Failsafe-to-user on every error path: pricing calculator failure → swap gated on billable > 0; previous-msg-time read failure → attribution_unavailable → bills y; NoOpService path (replay/tests) → bills y. No path bills more than the no-CAP world.
  • Promo parity: CalculateActualCost and CalculateColdCacheCost both pipe through applyOpusFastPromoCostUSD, so non-decoupled calls bill identically to the legacy calculateCostUSD path even during promo windows.
  • Shared calculator: NewPricingCalculator is memoized once per process via sync.Once; reused for every attribution.

13. Failure modes (fail-safe-to-user)

If anything in the attribution path errors out — DB unavailable, query timeout, pricing calculator hiccup — we fall back to billing the user at y. The user is never overcharged because of a bug.

FailureBehavior
BigTable SavePing fails after a ping fired Ping cost incurred (Anthropic charged), but no audit row persisted. Next user call won't see the ping → bills at y. Margin opportunity lost on that call. Logged from cap/workflow.go as a non-fatal warning.
BigTable client nil (NoOp path) SavePing early-returns nil; LoadCapPings returns empty. All attributions return no_qualifying_ping → bill y. Hit by replayer/tests, and by misconfigured pods where the CAP_BIGTABLE_* env vars are missing — verify with the "CAP pings BigTable store initialized" startup log line.
LoadCapPings BigTable range scan fails at setup cap.Service.LoadContext logs a non-fatal warning, leaves setup.Cap.Pings empty. PostHook sees empty list → no_qualifying_ping for the workflow's lifetime → bill y. Parallel goroutine is panic-safe via ego.SafeExec.
GetPreviousAssistantMessageTime SQL fails ResolveAttribution returns failsafe: BillableCostUSD = ActualCostUSD, EligibilityReason = attribution_unavailable. Logged + metric counted under that reason for dashboard visibility.
GetHeadMessageUsage SQL fails (global cache baseline) Baseline stays zero. DecideEligibility downgrades any would-have-decoupled call to no_global_cache_baseline → bills y. Conservative — refuses to decouple without a measured baseline to avoid over-claiming z.
NoOpService wired (no Store) All methods short-circuit cleanly: SavePing noop, LoadContext empty, ResolveAttribution returns failsafe. Replaces the previous nil-guard pattern at every callsite — no defensive checks needed in elite/cap workflow code.
Pricing calculator init error (z = 0 while y > 0) CAP swap is gated on BillableCostUSD > 0 — so a partial pricing failure that produces z = 0 on the decoupled path can no longer underbill. The original (working) y stays on the request. Reported by attribution-metrics counter under reason="decoupled" with margin_usd_sum = 0.
Temporal activity retry on activityPostHook Attribution result replays deterministically from workflow history. User billed exactly once at the resolved value. No double-charge possible.
Cortex pod missing CAP_BIGTABLE_* env vars (deployment rollover) BT client is nil → SavePing silently noops despite "CAP ping persisted" log firing (known cosmetic issue in current log line). Subsequent LoadCapPings from a healthy pod returns zero rows → bills y. Operational symptom: cap_attribution_total{reason="no_qualifying_ping"} stays high while cap_refresh_total{status="success"} climbs. Mitigation: verify all pods carry the env vars before a rollout.
