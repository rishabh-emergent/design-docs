Truncate-or-Summarize Gate at the Context Window Boundary

Status: Design · Scope: cortex elite workflow + xcontext · Builds on: range-based-truncation-opus-4-7 (Tier 1) · Gating: rides the same experiment as range-based truncation.

TL;DR

Once a session's context climbs past the top of the operating window (W_high, the existing range_threshold.upper), cortex today fires its hard-floor truncation every step. When the surviving context is itself already above W_high, those truncations reclaim almost nothing yet still re-process the full prompt — wasted work that delays the summarization actually needed.

This change adds a boundary gate: at the first step-finish after context exceeds W_high, it computes what the pending truncation would actually remove and chooses once — truncate if that removal is effective (≥ min_removal_frac of context and lands back inside the window), otherwise summarize immediately. The decision is made once per window-exit and only where range-based truncation is already on.

The one-line invariant

At or below W_high behaviour is byte-for-byte unchanged. Above W_high, the gate replaces "always truncate" with "truncate only while truncation still works; otherwise summarize."

2. How context is bounded today

Cortex keeps the elite agent loop's context within the model window using two independent mechanisms, evaluated in this order on every iteration (core/workflows/elite/flow.go):

#MechanismWhat it doesTrigger today
1Truncation (squash, bulk_checkpoint) Replaces old tool-result content with short stubs; cheap, no LLM call. For range-gate agents: cache-aware range gate in [lower, upper), plus an unconditional hard floor at frac ≥ upper.
2Summarization (auto-compact) Summarizes the conversation into one message and resets to a small context; an LLM call. auto_compact.threshold (the "0.35" of the window).

The range gate (Tier 1, shipped) is a pure function — rangeThresholdOutcome in core/xcontext/tiered.go:

// core/xcontext/tiered.go  (today)
switch {
case frac >= p.Upper:                                  // S > W_high
    return rangeThresholdHardFloor                     // ← truncate, unconditionally, every step
case frac >= p.Lower && elapsed >= deadline:           // in-window, cache dead
    return rangeThresholdCacheInvalidation             // ← truncate
default:
    return rangeThresholdNo
}

Three different numbers live near each other and are easy to confuse — we keep them distinct throughout: range_threshold.upper = 0.325 (W_high, the hard floor), overflow_threshold = 0.34 (a hard stop used only when auto-compact is off), auto_compact.threshold = 0.35 (today's summarization point).

3. The problem

The hard-floor arm returns rangeThresholdHardFloor for every finish where frac ≥ upper. With bulk_checkpoint, each fire only re-truncates the tool results that arrived since the last cutoff — often a few KB. Meanwhile the preserved tail (the most recent preserve_last_n tool-result messages plus everything after them) is carried in full and re-sent to the model on every step.

When that preserved tail is itself larger than W_high, truncation can never bring context back inside the window — there is nothing left that truncation is allowed to remove. The session lingers at an oversized context, paying to re-process the whole prompt each step, until it finally drifts up to auto_compact.threshold and summarizes. The work spent truncating in between bought almost nothing.

The anti-pattern we are removing

Truncate → reclaim 1%, still above the window → next step truncate → reclaim 1% → … → eventually hit 0.35 and summarize anyway. The system should have recognised at the first step that truncation was futile and summarized then.

4. The solution

Insert a decision at the moment the hard floor would fire. At the first finish where S > W_high, run the pending truncation for real on the current messages, measure how much it removes, and choose once: if the removal is effective (removes ≥ min_removal_frac of context and lands ≤ W_high) the hard-floor truncation proceeds as today; otherwise summarization fires immediately. An armed flag makes the choice exactly once per window-exit.

The gate intercepts exactly one thing — the hard-floor truncation at frac ≥ upper. The blue window keeps today's cache-aware behaviour; 0.35 stays only as a backstop.

The gate reuses both existing mechanisms verbatim. TRUNCATE = today's bulk_checkpoint hard-floor squash. SUMMARIZE = today's maybeApplyAutoCompact, simply allowed to run below its own 0.35 trigger when (and only when) the gate selects it. No new truncation or summarization machinery is written.

6. The rule (decision logic)

6.1 The state machine

Evaluated once per iteration finish, only when the gate is active (range_threshold present and min_removal_frac > 0):

The hard-floor truncation is itself gated behind armed, so "S > W_high but already decided" falls into SUPPRESS with no dedicated branch. That state is practically unreachable: any TRUNCATE or SUMMARIZE shrinks the next call's usage, which re-arms on the following finish.

6.2 Effectiveness — measured, not estimated (PRD R6)

The decision rests on what truncation actually removes. We run the real transform — xllm.ApplySquash (pure, no mutation, no I/O, safe in workflow code) — with the pending bulk_checkpoint cutoff on the current messages (which already reflect prior truncations), then compute:

removal_frac = (est_pre − est_post) / est_pre


post_trunc  = S × (est_post / est_pre)


effective   = removal_frac ≥ min_removal_frac  AND  post_trunc ≤ W_high

Why the ratio is in estimate-space but post_trunc is in real tokens

S is the projected next-prompt size (CompactionTokenCount = previous input + output + trailing); a post-truncation size can only come from cortex's char/4 estimator (no tokenizer inside deterministic workflow code). Mixing the two incoherently would let the effectiveness verdict disagree with the trigger at the boundary. So we take removal_frac as a pure ratio of estimator output (the estimator's systematic bias cancels) and scale the real S by the survival ratio to get post_trunc — keeping it in the same real-token units as W_high. Trailing tokens that the estimator misses appear on both sides of the ratio and cancel. A true token count would need an LLM round-trip: synchronous latency + replay nondeterminism, a non-starter under Temporal.

If the cutoff cannot advance (nothing new to truncate), the transform removes ~0 ⇒ removal_frac → 0 < min_removal_frac ⇒ SUMMARIZE. That is the desired outcome: when truncation is exhausted, summarize.

6.3 Reference pseudocode → cortex mapping

ON each step FINISH (gate active):
  S = CompactionTokenCount(messages, systemPrompt, lastUsage)   // projected NEXT prompt size
  if S <= W_high:                                               // W_high = range_threshold.upper * ctxWindow
      armed = true                                              // re-arm; existing cache-aware squash runs
      action = PROCEED
  elif armed:
      removal_frac, post_trunc = measure(pending bulk truncation on current msgs)
      effective = removal_frac >= min_removal_frac AND post_trunc <= W_high
      action = effective ? TRUNCATE : SUMMARIZE
      armed = false                                             // tentative; caller re-arms if action no-ops
  else:
      action = SUPPRESS

CALLER (elite/flow.go) dispatches:
  TRUNCATE  → forceReason = ReasonRangeThresholdHardFloor       // bypass the activity's input-only check
  SUMMARIZE → if auto_compact disabled: fall back to PROCEED + stay armed
              else:                     forceSummarize = true; allow image-only squash if MaxImages exceeded
  SUPPRESS  → allow image-only squash if MaxImages exceeded; otherwise no-op
  AFTER squash + compact ran: if the chosen action produced no effect, re-arm.

All on a 1M window: W_low=140K, W_high=325K, min_removal_frac=0.10, auto_compact.threshold=350K. Assume est_pre ≈ S for readability.

8. Data flow

The gate is a pure call returning one of four actions; the loop dispatches to the existing truncate / summarize paths. The new wiring is the dispatch, the force flag into auto-compact, an imageOnly flag on the squash request (lets image-overflow cleanup run without re-firing the hard floor under Suppress / Summarize), and a forceReason signal that bypasses the squash activity's input-only token check when the gate picks Truncate.

9. State model

The only new state is the armed boolean, held in the in-memory workflow State (core/workflows/elite/state.go) — the same place pendingSquashMargin and the summary-computation future already live. There is no new database column, BigTable family, or Redis key.

On resume / ContinueAsNew, State is rebuilt and armed resets to its default. This is harmless and self-healing: a session always starts well below W_high, so the first finish re-arms before any window-exit can occur; at worst a freshly-resumed run re-evaluates the gate one extra time. We deliberately do not persist armed — the decision is cheap to recompute and persisting it would add storage for a flag that converges on its own.

Default value: armed = true (set explicitly at State construction). Even if it defaulted to the zero value (false), the re-arm on the first low-S finish would correct it before W_high is ever crossed.

10.4 Loop dispatch — core/workflows/elite/flow.go

// runSquash:        used when the squash is gated by the existing input-only predicate (Proceed / image-only)
// forceSummarize:   passes through maybeApplyAutoCompact, bypassing its 0.35 NeedsCompact check
// imageOnlySquash:  zeros RangeThreshold + CompactionThreshold in the squash request so the
//                   activity can ONLY return ReasonImageOverflow (hard floor can't re-fire)
// forceSquashReason: forces the activity's decideSquashReason to return this string, bypassing the
//                    input-only token check so a BoundaryTruncate decision actually fires
runSquash         := true
forceSummarize    := false
imageOnlySquash   := false
forceSquashReason := ""
boundaryAction    := xcontext.BoundaryProceed
if setup.IsBoundaryGateActive() {
    d := decideBoundary(setup, state, messages)
    boundaryAction = d.Action
    state.SetBoundaryArmed(d.Armed)        // tentative; re-armed below if the action no-ops
    log.Info("boundary gate", "action", d.Action.String(), "s", d.S, "w_high", d.WHigh,
        "post_trunc", d.PostTrunc, "removal_frac", d.RemovalFrac, "armed", d.Armed)
    switch d.Action {
    case xcontext.BoundaryTruncate:
        forceSquashReason = xcontext.ReasonRangeThresholdHardFloor
    case xcontext.BoundarySummarize:
        if setup.IsAutoCompactOff() {
            // Misconfigured: act like Proceed and stay armed so we don't get stuck.
            state.SetBoundaryArmed(true)
            boundaryAction = xcontext.BoundaryProceed
        } else {
            runSquash       = xcontext.NeedsImageOverflowSquash(setup.SquashPolicy, messages)
            imageOnlySquash = runSquash
            forceSummarize  = true
        }
    case xcontext.BoundarySuppress:
        runSquash       = xcontext.NeedsImageOverflowSquash(setup.SquashPolicy, messages)
        imageOnlySquash = runSquash
    }
}

shouldSquash := forceSquashReason != "" ||
    (runSquash && needsSquashing(setup.SquashPolicy, setup, state, messages, squashEvalTime))
if shouldSquash {
    squash = w.applySquashing(exec, req, setup, agent, state, messages, squashEvalTime,
        imageOnlySquash, forceSquashReason)
    if squash != nil { messages = squash.Messages }
}

// ... auto-compact (force=forceSummarize) runs next ...

// Re-arm the boundary gate when the chosen action produced no effect, so the next iteration retries.
switch boundaryAction {
case xcontext.BoundaryTruncate:
    if squash == nil || (squash.Stats.TruncatedCount == 0 && squash.Stats.ImagesRemoved == 0) {
        state.SetBoundaryArmed(true)
    }
case xcontext.BoundarySummarize:
    if !compactOutcome.Compacted { state.SetBoundaryArmed(true) }
}

Two-stage armed: the gate's BoundaryDecision.Armed is the tentative new value (false on Truncate/Summarize). After both arms have run, the caller re-arms if the chosen action did nothing — covering misconfigured runs, cancelled summaries, and degenerate cases where the activity returns no-op.

10.5 Forced summarize — core/workflows/elite/compact.go

// maybeApplyAutoCompact gains a `force bool`. One guard changes:
proceed := status.NeedsCompact || (force && status.HasEnoughMessages)
if !proceed { return result }     // was: if !status.NeedsCompact { return result }

Everything downstream — pre-compact hook, summary acquisition, save, thread reload, optional handoff — is untouched. force still respects HasEnoughMessages so a session with ≤ last_n messages can't be asked to summarize nothing.

10.8 Persistence — internal/pgsql/agents.go

The jsonb writer/reader for the agent config already round-trips range_threshold; the new field rides along inside the same object. Confirm the marshal/unmarshal covers min_removal_frac (one line each) and regenerate mocks if any querier signature is touched (none expected).

10.9 Opt-in — experiment patch (no YAML in repo)

Range-based truncation is enabled purely by the structural presence of range_threshold, patched in via the existing Unleash experiment — there is no boolean flag and no committed agent YAML. The gate rides the same experiment, which now also:

  • adds range_threshold.min_removal_frac: 0.10;
  • ensures auto_compact is enabled (compact_near_overflow, threshold 0.35, a last_n, and a summary_prompt_name) so the SUMMARIZE arm can fire;
  • leaves summary_computation_threshold unset.

Because the gate keys off min_removal_frac > 0, the code ships dark: until the experiment adds the field, even range-truncation jobs keep today's unconditional hard floor. Flipping the field on is the entire rollout switch.

Paste-ready LogQL:

# How often the gate picks each arm (tune min_removal_frac from this ratio)
sum by (action) (
  count_over_time({app="cortex-agentsdk"} |= "boundary gate" | logfmt [1h])
)

# Sessions where SUMMARIZE fired below 0.35 (the new behaviour)
{app="cortex-agentsdk"} |= "boundary gate" | logfmt | action="BoundarySummarize" | s < 350000

# Suppress events (should be rare; a spike means summarize is being deferred a lot)
{app="cortex-agentsdk"} |= "boundary gate" | logfmt | action="BoundarySuppress"

# How often the chosen action no-ops — high = misconfigured agent or activity errors
{app="cortex-agentsdk"} |= "boundary gate re-armed"

# Misconfigured summarize fallbacks (should be zero in steady state)

The initial 0.10 / 0.325 / 0.14 values are starting points (PRD: "require validation and tuning against production behavior"). The action split above is the primary tuning signal — if SUMMARIZE dominates immediately at the boundary, raise min_removal_frac or re-examine preserve_last_n.
