Range-Gate Margin Capture via Billing Decoupling

Sibling design to CAP Margin Capture. Assumes cortex#1474 (range-based truncation) and the Redis-anchor follow-up are live.

1. TL;DR (for the impatient)

When the range-gate fires range_threshold_cache_invalidation, it preempts a future cache-write that the user would otherwise have paid for. The user’s next LLM call lands on a truncated prefix that costs Anthropic less.

This design captures that savings as Emergent margin instead of passing it silently to the user. On the first LLM call after such a squash, activityPostHook swaps Request.CostUSD from the actual cost y to the as-if-no-range-gate cost z, exactly mirroring CAP’s billing-decoupling mechanism.

margin = z − y    (x = 0 — range-gate has no ping cost)

Scope deliberately tight: one new file (core/xcontext/margin.go), one new state field, one new branch in activityPostHook, three new OTel counters, one experiment flag. No new package, no new Temporal activity, no new DB column, no rename of core/cap/.

2. Prerequisites

This design assumes both of these are deployed:

  1. cortex#1474 — range-based truncation Tier 1. The RangeThresholdPolicy gate fires range_threshold_cache_invalidation when in-range and the prompt cache is predicted dead, range_threshold_hard_floor at upper unconditionally. ApplyAndPersistSquashResult.Reason carries the attribution string. elite.squash.cache_creation_avoided_tokens is already emitted as a metric.
  2. Redis cross-run anchor (cortex#1479). state.lastLLMCallTime is seeded from the Redis-backed cross-run anchor at workflow start, so the gate actually fires on the first iteration of a resumed run. Without this, the gate misses the most lucrative case and margin capture has nothing to capture.

If either is rolled back, this feature self-disables (the gate either never fires or fires on the wrong call — both observable in the fail-safe metrics).

3. The margin leak

From the range-based truncation design: in the cache-invalidation arm, the range-gate squashes preemptively at, say, 250k tokens because it predicts the cache is about to TTL-die. Without the gate, the user would have paid:

  • cache_create on the full ~250k prefix at the moment of natural TTL expiry, then
  • cache_create again on the truncated prefix when prompt eventually hits 275k.

With the gate, the user pays only the second cache_create — on the truncated prefix. The first one (full prefix) is the “avoided” event.

Today, the savings flow straight through. The user’s cost_usd reported to emergent reflects only the truncated cache_create. We pay Anthropic the same. Emergent earns nothing on this optimization. The PRD’s explicit ask:

“Deduction: We should deduct the cost that user would have incurred for the cache write … in case of invalidation at each of the call we help user reduce costs.”

Same shape as CAP’s leak: an optimization that reduces our spend with Anthropic also reduces user’s spend with us, when we could instead hold the user’s price flat and pocket the difference.

4. The decoupling idea (the fix)

Charge the user as if the range-gate never fired. Pay Anthropic the actual reduced amount. Margin per qualifying call = difference.

Concretely, on the first LLM call after a qualifying squash:

  1. Compute y exactly as today (calculateCostUSD on the call’s real Usage).
  2. Compute z by taking the same Usage and adding the avoided_cache_creation_tokens to CacheCreationInputTokens, then re-pricing. This is the only new math.
  3. In activityPostHook: Request.CostUSD = z; Request.AccumulatedCost += (z − y); mirror the delta into workflow state so iter N+1 carries forward.

The PostHook payload contract to emergent is unchanged. The only thing that changes is what numbers fill cost_usd / accumulated_cost — identical to CAP.

5. Vocabulary

Reuses CAP’s notation so cross-doc reading is cheap. There’s no x here because the range-gate doesn’t make its own LLM call.

y — actual-cost
What Anthropic charges us for the post-squash LLM call. Priced from the call’s real Usage by xllm’s pricing calculator. Same number that calculateCostUSD already produces today.
z — as-if-no-range-gate
What the call would have cost if the gate hadn’t fired. Constructed by adding avoided_cache_creation_tokens (from the squash event) to Usage.CacheCreationInputTokens, then pricing. z ≥ y, always.
avoided_cache_creation_tokens
The number of cache-create tokens the user did not pay. Computed at squash time as pre_squash_cached_prefix − post_squash_cache_create — see §6. Distinct from PR 1474’s metric-only over-approximation; we tighten the formula for billing.
range-gate decoupled call
An LLM call that meets every attribution gate. User pays z, Anthropic gets y, Emergent margin = z − y.

margin = z − y

5b. Sources of truth at squash time

The margin math depends on two values being correct at the moment the squash fires:

  1. Elapsed since last LLM call / CAP refresh — for the gate-fire decision (PR 1474 logic).
  2. Cached-prefix size at the moment of squash — for the PreSquashCachedPrefix baseline used by margin attribution.

Both must work for mid-run squashes (squash fires several iterations into an active workflow run) and resume squashes (squash fires on iteration 1 of a fresh workflow run after the user sent a follow-up message). The table below maps each value to its source in each case:

ValueMid-run (iter N>1, same workflow run)Resume (iter 1 of new workflow run)
state.lastLLMCallTime (for elapsed) flow.go:679 updates after every LLM call; cap.go:85 updates after every successful CAP refresh. Always fresh. Seeded at workflow start from the Redis cross-run anchor (cortex#1479). Falls back to exec.Now() when Redis is unavailable — safe (gate cannot fire on the first iteration in that case).
state.LastUsage (for PreSquashCachedPrefix) flow.go sets this from each LLM response’s Usage before the next iteration. state.LastUsage = the immediately prior LLM call’s Usage in this run. setup.go:329 loads from loadResult.LastUsage, which threads.go:69-76 populates as the most-recent assistant message’s Usage from the DB. So state.LastUsage on iter 1 of a resumed run = the prior workflow run’s last LLM call’s Usage. Verified in prod — see §7 Scenario A.

The design uses state.LastUsage.CacheCreationInputTokens + state.LastUsage.CacheReadInputTokens as PreSquashCachedPrefix. This is correct in both cases because both write paths flow into the same state.LastUsage field.

Why this beats the “use Redis for the Usage too” alternative

One could imagine also persisting Usage to Redis on every LLM call (alongside the timestamp). It’s unnecessary: the DB already stores Usage on every assistant message (threads.go reads it back on resume); the in-process state.LastUsage covers mid-run. Adding Redis for Usage would duplicate the DB and create new consistency questions. Skip it.

6. The attribution rule

We do not bill at z on every squash. Same philosophy as CAP: false positives are visible (overcharging); false negatives are silent. When in doubt, bill at y.

An LLM call is range-gate-decoupled iff all of these hold:

#GateRationale
1 state.pendingSquashMargin != nil A squash fired in this iteration. The pending event is set when the squash activity returns and cleared at the end of the same iteration’s PostHook.
2 Reason is range_threshold_cache_invalidation Only this reason corresponds to a real avoided cache_create. range_threshold_hard_floor defers the legacy 275k squash but does not avoid a cache_create (cache may have been alive). image_overflow and legacy_threshold are not range-gate optimizations.
3 Provider is Anthropic (provider_clamp) Cache-write pricing semantics are Anthropic-specific. Other providers’ cache models differ and the math would not apply.
4 Experiment flag RangeGateMarginEnabled is on Gradual rollout via the existing elite-experiments client. See §13.
5 avoided_tokens > 0 after subtracting post-squash cache_create If the post-squash prefix is larger than the pre-squash cache (unusual but possible with image squash interleavings), there is no avoidance.
6 y > 0 AND z > y Pricing must succeed for both. If either returns 0 or z ≤ y (math sanity), no swap. Mirrors CAP’s BillableCostUSD > 0 guard.

Computing avoided_tokens precisely

PR 1474’s metric uses an approximation that over-counts (it equates avoided to the full pre-squash cached prefix). For billing, we need a tighter formula because two things only become known after the post-squash LLM call returns:

  1. The post-squash cache_create size (how big the new prefix actually was).
  2. The post-squash cache_read size, which reveals the surviving baseline — the part of the prefix (typically system_prompt + tool definitions) that escapes TTL invalidation because it sits at a higher level in the cache hierarchy. This portion would not have been re-paid as cache_create even without the squash, so it must be subtracted from the savings claim.

So avoided_tokens is finalized at PostHook time, not at squash time:

// At squash time (xcontext.ApplySquash):
pendingMargin.PreSquashCachedPrefix = lastUsage.CacheCreationInputTokens +
                                       lastUsage.CacheReadInputTokens

// At PostHook time, using the current LLM call's Usage:
postSquashCacheCreate := currentUsage.CacheCreationInputTokens
postSquashCacheRead   := currentUsage.CacheReadInputTokens
avoided := pendingMargin.PreSquashCachedPrefix - postSquashCacheCreate - postSquashCacheRead
if avoided ≤ 0 { /* gate 5 fails, fall back to y */ }

Equivalent reading: how many tokens of context did we truncate that would have been cache_create on the next call. The formula reduces to PreSquashCachedPrefix − PostSquashCachedPrefix when expressed as prefix delta, which is the physical truth of what changed.

Why subtract cache_read too

Anthropic’s cache hierarchy keeps the system_prompt + tools prefix alive longer than the body messages prefix. In a TTL-invalidation event, typically only the body dies; the baseline (~28k for a mature Opus 4.7 agent) survives. We observed this directly in prod — see §7 Scenario A where a 4m16s gap dropped cache_read from 269k to 28k while cache_create jumped to 245k. The 28k baseline was not avoided by the squash — it would have been cache_read either way — so it doesn’t count toward margin.

Why we keep PR 1474’s metric AND the billing formula

PR 1474’s elite.squash.cache_creation_avoided_tokens is a coarse health signal — it answers “how often does the gate fire and how big are the squashes?”. The billing formula above is the conservative number used to compute the actual z − y swap. They diverge by PostSquashCacheCreate + PostSquashCacheRead for the same squash event. Both are correct for their respective purposes.

8. Data flow

Inside one iteration of runIteration:

1. squashEvalTime := exec.Now()
2. if needsSquashing(...) {
       result := applySquashing(...)
       if result.WasApplied && isRangeGateCacheInvalidation(result.Reason) {
           state.SetPendingSquashMargin(&xcontext.PendingMarginEvent{
               Reason:                 result.Reason,
               PreSquashCachedPrefix:  result.PreSquashCachedPrefixTokens,
               Model:                  setup.CurrentModel,
               Provider:               setup.CurrentProvider,
               FiredAt:                squashEvalTime,
           })
       }
   }
3. LLM call on the truncated prefix
4. tool execution
5. BuildPostHookRequest → hookReq.CostUSD = calculateCostUSD(model, usage) // y
6. ExecuteActivity(ActivityPostHook, PostHookActivityInput{
       Request, CapEnabled, CapPings, Iteration, PendingSquashMargin: state.PendingSquashMargin(),
   })
   // inside activityPostHook:
   //   - CAP attribution (existing) computes cap_delta
   //   - if input.PendingSquashMargin != nil && gates pass:
   //         rangeGateAttribution := xcontext.ResolveMarginAttribution(...)
   //         rg_delta = rangeGateAttribution.AsIfCostUSD - rangeGateAttribution.ActualCostUSD
   //         Request.CostUSD = y + cap_delta + rg_delta
   //         Request.AccumulatedCost += cap_delta + rg_delta
   //         (clears pending event via result.SquashMarginConsumed = true)
   //   - agentSvc.PostHook(Request)
7. state.AddCost(result.CostDiff)  // existing CAP path; covers both deltas
8. if result.SquashMarginConsumed { state.ClearPendingSquashMargin() }

Key property: the pending event lives exactly one iteration. If the LLM call errored before PostHook fired, the pending event is still alive next iteration and the next successful PostHook consumes it. If iter N+1’s squash also fires before that PostHook (unusual), the event is overwritten — the older opportunity is lost on purpose (we can’t bill two deltas to one call).

Estimate: ~150 production lines, ~150 test lines. One new file. No new package. No new Temporal activity.

9.1 core/xcontext/margin.go (NEW, ~80 lines)

Pure types and one resolver function. Lives next to the squash decision in the same package. Imports cap.NewPricingCalculator + cap.CalculateActualCost for pricing — see §10.

// core/xcontext/margin.go
package xcontext

import (
    "context"

    "cortex/core/cap"   // pricing primitives reused; no other cap dependency
    "cortex/pkg/agentsdk/io"
    "github.com/emergentbase/xllm"
)

// PendingMarginEvent is recorded when a range-gate cache-invalidation squash
// fires and consumed by the next PostHook on the same iteration.
type PendingMarginEvent struct {
    Reason                  string  // always range_threshold_cache_invalidation when set
    PreSquashCachedPrefix   int     // last call's cache_creation + cache_read
    Model                   string
    Provider                string
}

// MarginInput is the resolver input. Workflow-side state extracts these
// fields and hands them in; the resolver does pure math + pricing.
type MarginInput struct {
    Pending           *PendingMarginEvent
    CurrentUsage      *io.TokenUsage
    ProviderClamped   bool   // experiment + provider gate evaluated by caller
    ExperimentEnabled bool
}

// MarginAttribution is the resolver output. Decoupled=false means “bill at y”.
type MarginAttribution struct {
    Decoupled         bool
    ActualCostUSD     float64  // y
    AsIfCostUSD       float64  // z
    AvoidedTokens     int
    EligibilityReason string   // skipped reasons or "decoupled"
}

const (
    EligibilityDecoupled              = "decoupled"
    EligibilityNoPendingEvent         = "no_pending_event"
    EligibilityReasonNotEligible      = "reason_not_eligible"
    EligibilityExperimentDisabled     = "experiment_disabled"
    EligibilityProviderNotAllowed     = "provider_not_allowed"
    EligibilityAvoidedNonPositive     = "avoided_non_positive"
    EligibilityPricingFailed          = "pricing_failed"
    EligibilityMathSanityFailed       = "math_sanity_failed"
)

func ResolveMarginAttribution(ctx context.Context, in MarginInput) MarginAttribution {
    if in.Pending == nil { return failsafe(EligibilityNoPendingEvent, 0, 0) }
    if in.Pending.Reason != ReasonRangeThresholdCacheInvalidation {
        return failsafe(EligibilityReasonNotEligible, 0, 0)
    }
    if !in.ExperimentEnabled { return failsafe(EligibilityExperimentDisabled, 0, 0) }
    if !in.ProviderClamped   { return failsafe(EligibilityProviderNotAllowed, 0, 0) }

    avoided := in.Pending.PreSquashCachedPrefix -
               int(in.CurrentUsage.CacheCreationInputTokens) -
               int(in.CurrentUsage.CacheReadInputTokens)  // subtract surviving baseline
    if avoided ≤ 0 { return failsafe(EligibilityAvoidedNonPositive, 0, 0) }

    calc := cap.NewPricingCalculator(ctx)
    y    := cap.CalculateActualCost(calc, in.Pending.Model, in.CurrentUsage)
    if y == 0 { return failsafe(EligibilityPricingFailed, 0, 0) }

    asIf  := cloneWithAddedCacheCreate(in.CurrentUsage, avoided)
    z, err := calc.Calculate(in.Pending.Model, asIf)
    if err != nil || z.TotalCost == 0 {
        return failsafe(EligibilityPricingFailed, y, 0)
    }
    if z.TotalCost ≤ y { return failsafe(EligibilityMathSanityFailed, y, z.TotalCost) }

    return MarginAttribution{
        Decoupled:         true,
        ActualCostUSD:     y,
        AsIfCostUSD:       z.TotalCost,
        AvoidedTokens:     avoided,
        EligibilityReason: EligibilityDecoupled,
    }
}

10. No duplication with CAP

We reuse two entry points from core/cap/:

cap.NewPricingCalculator(ctx)  // returns *xllm/pricing.Calculator wired with cortex defaults
cap.CalculateActualCost(calc, model, usage)  // prices a real Usage; same fn already used for y

We do not reuse cap.CalculateColdCacheCost — its swap shape is different (cache_read → cache_create, accounting for a global cache baseline). The range-gate swap (cache_create += avoided) is a separate operation, performed inline inside ResolveMarginAttribution via a one-line clone of Usage. No new helper added to cap.

Why not extract pricing into a neutral core/billing/?

Tempting but out of scope. The two entry points we reuse are already thin wrappers over xllm/pricing. Extracting them buys consistency at the cost of one rename PR touching every CAP call site. Defer until a third consumer needs them.

We also do not reuse cap.Attribution, cap.EligibilityDecision, or any cap attribution constants. Those carry CAP-specific fields (PingRequestID, GlobalCacheBaseline). Mixing them into a range-gate path would force the cap package to learn about squash events — the wrong layering. MarginAttribution has its own minimal shape (4 fields), independent of cap.Attribution.

11. Composition with CAP on the same call

A CAP-decoupled call and a range-gate-decoupled call can happen on the same LLM call (rare but possible: CAP refresh kept the cache alive through a long tool execution, then the next iteration’s prompt re-entered the range and the cache-invalidation arm fired). Both deltas need to stack.

Order inside activityPostHook:

  1. Existing CAP block computes cap_delta = cap.BillableCostUSD - cap.ActualCostUSD. If non-zero, Request.CostUSD = cap.BillableCostUSD; AccumulatedCost += cap_delta.
  2. New squash-margin block reads the current Request.CostUSD (which may already be z_cap) and the original Usage. Computes y from Usage directly (not from Request.CostUSD, which may be post-CAP), then z_squash, then rg_delta = z_squash - y. Adds rg_delta on top.

Both deltas reference the same y baseline (the call’s real cost). They are additive because they capture different mechanisms — CAP captures “cache stayed alive past TTL”; squash margin captures “cache was killed to avoid a future cache_create”. Mathematically disjoint.

final_billable = y + cap_delta + rg_delta
                = y_actual + (z_cap - y) + (z_squash - y)
                = z_cap + z_squash - y

result.CostDiff = cap_delta + rg_delta; state.AddCost(result.CostDiff) mirrors both into workflow state.

Bias toward under-billing

Every check is “bill at y if anything looks off”. We’d rather miss margin than overcharge. The skipped-metric labels let us see exactly how much we’re leaving on the table by reason.

14.2 Cross-reference with PR 1474’s squash metrics

Existing: elite.squash.cache_creation_avoided_tokens{reason=range_threshold_cache_invalidation} tells us “how often does the gate fire and how big is the squash?” (using the loose formula).

New: squashmargin.attribution.avoided_tokens_total tells us “how many of those squashes were billable and what was the tight avoided figure?”.

Their ratio is the conversion rate of squash events into captured margin — useful for tuning cache_ttl_guard or expanding the experiment cohort.

15. Non-goals

  • No backfill, no historical replay. Margin starts accruing from PR merge.
  • No emergent-side billing pipeline change. credit_manager.deduct(scaled_cost) reads cost_usd off the existing payload exactly as today.
  • No tier-2 (summarisation) margin in this PR. Out of scope; design generalises but the gates only admit range_threshold_cache_invalidation.
  • No new BigTable column family. Whole margin lifecycle lives in-workflow (squash → PostHook → cleared) so there is nothing to persist beyond the existing workflow history.
  • No per-user override surface. Per-agent opt-in plus the experiment flag are the only knobs.
  • No extraction of pricing into a neutral core/billing/. Reuse via direct import from core/cap/; defer the rename until a third consumer appears.
