1. Pehle samjho problem

Anthropic prompt cache ki 3 reality:

💸 Pricing fact

cache_read tokens = roughly 10% of input price.
cache_create tokens = roughly 1.25× input price.
Bada prompt + dead cache = bahut mehnga.

🏠 Old behavior

Squash threshold = 0.275 hard. Mtlb 275k tokens cross hote hi truncate. Cache alive ya dead, koi farak nahi. Agar cache alive tha to hum cheaply re-use kar sakte the, but truncation se hum khud cache invalidate kar dete the.

2. Concept — Range gate + Margin capture

Range gate band

Iss PR me ek range_threshold object aata hai agent YAML me:

range_threshold:
  lower: 0.200          # 200k tokens — cache-invalidation band start
  upper: 0.325          # 325k tokens — hard squash floor
  cache_ttl_guard: 15s  # cache_ttl (5m) − 15s = deadline

Decision logic (core/xcontext/tiered.go):

func rangeThresholdOutcome(p, ctxWindow, tokens, elapsed) reason {
    frac := tokens / ctxWindow
    deadline := p.CacheTTL - p.CacheTTLGuard
    switch {
    case frac >= p.Upper:
        return rangeThresholdHardFloor                // 325k+: always squash
    case frac >= p.Lower && elapsed >= deadline:
        return rangeThresholdCacheInvalidation        // 200k–325k AND cache dead: squash
    default:
        return rangeThresholdNo                              // keep cache alive
    }
}

Margin capture — y / z / margin

TermMeaning
yActual cost paid to Anthropic (squash hua, cache_create save ho gaya)
zAs-if cost — agar range gate NA hota, agent ko full cache invalidation pay karna padta
avoidedcache_create tokens jo "bachaye" — pre_squash_cached_prefix − post_cache_create − post_cache_read
marginz − y — Emergent ka extra revenue (kyunki user ko z bill ho raha hai)
Sanity guard: agar koi bhi sanity check fail ho jaye (avoided ≤ 0, pricing nil, z < y) to swap nahi hota → user ko original y hi bill hota hai. Fail-safe by design.
