In-Loop CAP Margin Capture

Extends cap-margin-capture. Captures margin on the LLM call following a long-running tool execution. Read the terminal-state design first — this doc assumes that vocabulary and machinery.

1. TL;DR (for the impatient)

CAP already keeps the prompt cache alive during long-running tool executions via in-loop refresh pings (core/workflows/elite/cap.go:capRefreshLoop). The firing side is shipped and battle-tested. But the attribution side — the part that re-bills the next LLM call at the as-if-cold-cache price z and captures the margin — is hard-gated to iter 1 of every workflow (activities.go:776, the IsFirstIteration short-circuit). That gate exists because the PrevAssistantMsgAt eligibility boundary is snapshotted from Redis at workflow setup and never advanced per-iter; running attribution on iter 2+ with a stale window would re-claim the same ping pool and double-count.

This change makes one targeted shift: delete the iter-1 gate, advance the eligibility window per-iter, and merge in-flight in_loop pings into the attribution input. Same z − y − x math, same Unleash experiment, same metrics. No new flag, no schema change.

Scope

  • Same flag as terminal-state. Reuses cap_enable_refresh; a patch on the same experiment toggles LongRunningMaxRefreshes > 0 to fire in-loop pings. If firing is on, attribution follows.
  • No new package, no new activity, no new metric. The existing cap.attribution.* histograms already carry a ping_source label (set from PingRecord.Source) — in-loop margin surfaces as a ping_source="in_loop" slice on the same Grafana panels.
  • 5 files changed, +20 / −10 lines net. Net code shrinks: deleting the IsFirstIteration gate removes more code than the per-iter advance adds.

2. Prerequisites

This design assumes a working understanding of the terminal-state CAP Margin Capture doc — specifically:

  • The y / z / x vocabulary (section 6 of the parent doc). Margin per decoupled call = z − y − x.
  • The attribution rule (parent section 7): decouple only when (a) elapsed since the previous elite call exceeds CacheTTL = 5min, AND (b) at least one cache-hit ping fired in the eligibility window (prevAt, callTime].
  • The Redis runtime-meta layout (parent section 11): last_assistant_msg_at is the boundary; CAP pings deliberately do NOT advance it.
  • BigTable ping_data column family as the persistence layer for all pings, in_loop and post_exit alike (parent section 10). Here we will not load in-flight in_loop pings from BigTable — we'll carry them in workflow state.

3. What's shipped vs the gaps

The firing side of in-loop CAP is mature. The gap is entirely on the attribution side, and it's smaller than it looks.

3.2 The two attribution gaps

#GapWhere it breaksSymptom
G1 In-flight in_loop pings are invisible to attribution. refreshCacheAnchorIfNeeded writes to BigTable via the activity, but the result is dropped on the floor — nothing is appended to any in-memory list the PostHook activity can read. elite/cap.go:73-93 Even with all filters removed, iter N+1 sees CapPings: setup.Cap.Pings (frozen at setup) → empty → no_qualifying_ping → bills y.
G2 Attribution is hard-gated to iter 1. The !input.Iteration.IsFirstIteration short-circuit was added precisely to prevent double-counting the same post_exit pool on later iters — at the cost of making in-loop margin uncapturable. elite/activities.go:776 Iter 2+ never run attribution; log line is cap attribution skipped reason=not_first_iteration.

Why G2 exists today (and why removing it is the right call now)

With PrevAssistantMsgAt frozen at workflow-setup load, the eligibility window for iter N is (Redis-loaded-time, iter_N_CallTime]. Both iter 1 and iter 2 would see the same post_exit pings in their windows → iter 2 would re-claim the margin iter 1 already claimed. The cheap defense was "only attribute on iter 1". That works for terminal-only CAP. It doesn't work for in-loop, because in-loop pings fire between iters, and only iter 2+ can claim them. The proper fix is per-iter window advance — which makes the gate redundant.

4. Vocabulary (delta)

No new vocabulary. We reuse y, z, x from the parent doc. One existing term is highlighted because it gains a new role:

eligibility window
For attribution at iter N: the time interval (prevAt, callTime] where prevAt is the previous elite LLM call's start time. Today (terminal-only): prevAt = Redis-loaded value, fixed for the workflow. After this change (unified): prevAt = iter (N−1)'s CallTime for iter ≥ 2; prevAt = Redis-loaded value for iter 1.
PingRecord.Source
Already exists: "post_exit" | "in_loop". Currently used as a hard filter (drop in_loop). After this change, used only as an audit label for metrics/logs — both sources participate in attribution equally.

5. The design

5.1 Two new fields on workflow state

// core/workflows/elite/state.go
type State struct {
    // ... existing fields ...

    lastAssistantMsgAt time.Time          // seeded from setup.Cap.LastAssistantMsgAt at workflow start;
                                          // advanced to thisIter's CallTime per elite LLM call;
                                          // read by PostHook builder as the previous-iter boundary.

    inLoopPings        []cap.PingRecord    // empty at workflow start;
                                          // appended after each successful refreshCacheAnchorIfNeeded;
                                          // merged into PostHookActivityInput.CapPings.
}

func (s *State) GetLastAssistantMsgAt() time.Time          { return s.lastAssistantMsgAt }
func (s *State) SetLastAssistantMsgAt(t time.Time)         { s.lastAssistantMsgAt = t }
func (s *State) GetInLoopPings() []cap.PingRecord          { return s.inLoopPings }
func (s *State) AppendInLoopPing(p cap.PingRecord)         { s.inLoopPings = append(s.inLoopPings, p) }

5.2 The read-advance-merge pattern at every PostHook build site

// core/workflows/elite/flow.go (build site for PostHookActivityInput, ~line 950)

// Capture before advance — this is what attribution uses as "previous iter".
prevAt := state.GetLastAssistantMsgAt()
state.SetLastAssistantMsgAt(p.LLMCallStartedAt)

// Merge in-flight in_loop pings on top of the setup-loaded post_exit pings.
mergedPings := append(append([]cap.PingRecord{}, setup.Cap.Pings...), state.GetInLoopPings()...)

input := PostHookActivityInput{
    Request:    *postHookReq,
    CapEnabled: setup.Cap.Enabled,
    CapPings:   mergedPings,
    Iteration: IterationContext{
        ...
        PrevAssistantMsgAt: prevAt,
        // IsFirstIteration: DELETED
    },
    PendingSquashMargin: state.PendingSquashMargin(),
}

Why a local for prevAt, not a third state field

The "previous" value is only meaningful at the precise instant we build PostHook input. Once we've used it, advancing the canonical lastAssistantMsgAt immediately makes the snapshot stale. Co-locating snapshot + advance in three lines (read, set, use) eliminates the class of bug where a future contributor adds a fourth read site and gets the wrong value. The cons of an explicit state.prevAssistantMsgAt sibling field are documented in ADR-0001.

5.3 Appending to the in-loop ping list

RefreshResult gains one field — FiredAt time.Time, captured by the activity at the moment it actually fires the ping. That's the only datum the workflow doesn't already know; everything else needed for a PingRecord is in the RefreshRequest that was sent in.

// core/cap/refresh.go (add to RefreshResult)
FiredAt time.Time `json:"fired_at"`

// core/workflows/cap/workflow.go (activityRefreshCacheAnchor — set on the result before return)
result.FiredAt = time.Now()

// core/workflows/elite/cap.go (refreshCacheAnchorIfNeeded, after activity succeeds)
state.AppendInLoopPing(cap.PingRecord{
    JobID:               req.JobID,
    SessionID:           setup.Agent.SessionID,
    RequestID:           refreshReq.RequestID,
    Source:              cap.PingSourceInLoop,
    Model:               setup.CurrentModel,
    CacheHit:            result.CacheHit,
    CacheReadTokens:     result.CacheReadTokens,
    CacheCreationTokens: result.CacheCreationTokens,
    CostUSD:             result.CostUSD,
    FiredAt:             result.FiredAt,
})

The activity's time.Now() is captured deterministically into the activity result, which Temporal records in workflow history. Replays of the same workflow see the same FiredAt on the same iteration. No replay nondeterminism.

What does NOT change

  • internal/pgsql/cap_pings.go:LoadCapPings keeps its source != post_exit filter on setup-time loads. This is intentional — in-loop pings only matter within the workflow that fired them; cross-workflow load only needs post_exit pings. We carry the in-flight ones in workflow state instead.

  • Existing failure modes from the parent doc all still apply with the same semantics. BigTable write failure on a ping → ping not persisted, but it is in state.inLoopPings for the rest of this workflow → margin attributed correctly. Better than terminal-state, where BT-write-failure means the ping is lost forever.

6. Data flow

Three iters of a single workflow showing the unified attribution path. iter 1 claims post_exit pings (from prior workflow); iter 2's tool execution generates in_loop pings; iter 3 claims them.

Same workflow, three iters, one unified attribution path. iter 1 picks the most recent post_exit ping (terminal-state semantics, unchanged). iter 2 picks the most recent in_loop ping (new). iter 3's window starts at iter 2's CallTime so the iter-2 pings are no longer in scope — no double-count. The pings stay in state.inLoopPings for the workflow's lifetime; only the window moves.

7. Race conditions and edges

Surfaced during design grilling. None require new code beyond the design above — all resolve via the per-iter window semantic.

#Race / edgeResolution
R1 Ping fires between iter-N LLM-call-start and PostHook build. Possible if response latency is high. FiredAt > iter-N's CallTime → ping falls into iter (N+1)'s window. Correct attribution; nothing lost.
R2 capRefreshLoop fires concurrently with iter-(N+1)'s LLM-call dispatch. If ping's FiredAt < iter-(N+1)'s CallTime → in iter-(N+1)'s window. If FiredAt > iter-(N+1)'s CallTime → in iter-(N+2)'s window. Either way: claimed by exactly one iter.
R3 capRefreshLoop sleeping during workflow exit. exec.Sleep returns context_cancelled; loop terminates cleanly. Identical to today's behavior.
R4 Continue-as-new boundary mid-tool. Elite CAN-rolls into a new workflow run; state.inLoopPings is lost in the transition. Acceptable: pings persisted in BigTable but won't be in-memory after CAN. The new run's iter 1 has setup.Cap.Pings with post_exit only (LoadCapPings filter); in-loop pings from the pre-CAN tool execution are effectively forgotten. Margin loss bounded by CAN frequency (rare; only adaptive-thinking path). Could revisit if CAN-during-tool becomes common — for now, accept.
R5 Subagent runs inside parent's tool. Parent's capRefreshLoop keeps firing against parent's prompt while the subagent has its own workflow + session. Subagent has separate sessionID; its pings have separate BigTable row keys. Parent's in-loop pings land on parent's state.inLoopPings and get claimed by parent's next iter. No cross-contamination.
R6 BigTable PUT fails after Anthropic call succeeds. Anthropic charged us; ping not persisted. But the workflow still calls state.AppendInLoopPing from the activity result — so in-flight attribution within this workflow still works. Only cross-workflow attribution (a future workflow's setup-time load) loses the ping. Better than terminal-state, where the same failure mode loses the ping entirely.
R7 Tool exceeds LongRunningMaxRefreshes. Pings stop firing while tool still runs; cache may die before iter N+1. PickCandidate requires CacheTTL proximity to CallTime — stale pings are rejected → no_qualifying_ping → bills y. Margin opportunity missed; no incorrect billing. Tunable via config.
R8 Profitability inversion: x > (z − y). Theoretical at LongRunningMaxRefreshes ≥ 8; not a concern at the default 3–4. Accepted risk per design decision: no gate. Watch cap_attribution_margin_usd_sum for negative values; tune LongRunningMaxRefreshes down if observed.
R9 Replay determinism on activity result. FiredAt is captured inside the activity (time.Now()), recorded in workflow history. Replay reads the same value from history → workflow's state.AppendInLoopPing appends the same record at the same position. Deterministic.
R10 Workflow versioning (TWD). In-flight workflows replay with old code (still has IsFirstIteration); new workflows have new code (no gate). No history-mismatch risk. Pinned versioning per cortex/CLAUDE.md "Workflow Versioning Policy". Field removal from IterationContext is safe — Go JSON unmarshalling silently drops unknown fields on the in-flight side if anything ever does cross-version.

func SumPingCosts(pings []PingRecord, after, upTo time.Time) float64 {

9. Failure modes (delta from parent doc)

All failure modes from the parent section 12 still apply unchanged. Two new ones, both fail-safe-to-user:

FailureBehavior
In-loop ping activity panics / errors mid-fire Activity returns Succeeded=false; workflow does not append to state.inLoopPings; no margin claimed on next iter. Anthropic charged for the partial work (negligible). Identical to today's behavior for this failure path.
FiredAt not populated by activity (bug / forgotten field) Zero time → PickCandidate's FiredAt.After(after) returns false → ping never eligible → no_qualifying_ping → bills y. Margin opportunity missed; no incorrect billing. Unit test covers this.
