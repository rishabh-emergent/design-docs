Range-Based Truncation for Opus 4.7 — Tier 1 Design

Date: 2026-05-22  ·  Status: Truncation only (Tier 1). Summarisation deferred to a follow-up stacked PR.  ·  Scope: Opus 4.7 farm agents on 1M context window  ·  PR: cortex#1474

What changed since v0.2

The earlier draft (v0.2) introduced a dedicated range_based_truncation block with twin specs (truncate_range / summarise_range) and a 3-value action enum. The implementation shipping in cortex#1474 instead bolts a single generic range_threshold field onto the existing Context type and reuses the existing squash dispatch. It's smaller, more idiomatic to cortex, and the same struct will plug into auto-compact for Tier 2 in a follow-up PR. This doc has been rewritten to match the shipping design. Summarisation (Tier 2) is out of scope for this design — it will get its own design doc when stacked.

1. TL;DR

Today, Opus 4.7 farm agents squash exactly when prompt_tokens crosses the 275k threshold. Anthropic's prompt cache is prefix-based with a 5-minute TTL. A TTL-driven invalidation that lands just before 275k forces a cache-write on the post-invalidation call, then a few calls later the squash fires and we pay a cache-write again on the new (truncated) prefix. We pay for cache invalidation twice.

This design adds a cache-TTL-aware range gate on top of the existing bulk_checkpoint squash. A single configurable range sits below the legacy hard threshold:

  • Truncation range [lower, upper] — recommended [0.200, 0.325] on a 1M context window (= 200k–325k tokens).
  • Inside the range, fires only when the cache is predicted to be invalid — i.e., elapsed since the last LLM call or CAP refresh is past cap.CacheTTL − cache_ttl_guard.
  • At or above upper, fires unconditionally — the hard floor.
  • Below lower, no fire.
  • Legacy threshold: 0.275 is replaced for opted-in agents — the range gate is the sole truncation trigger.

Hard contracts

  • Works whether CAP is on or off. The signal is elapsed = now − state.LastLLMCallTime. CAP refreshes already update LastLLMCallTime (any successful ping, whether the cache hit or not), so CAP and the range gate share a single source of truth without explicit plumbing.
  • Fully backward compatible. Agents without the new field see zero behaviour change. Existing threshold, overflow_threshold, auto_compact[], preserve_last_n, truncation_length keep their semantics for non-opted-in agents.

2. The problem we're solving

Anthropic's prompt cache is prefix-based with a 5-minute TTL. cache_read is ~12.5× cheaper than cache_creation. Cache-creation gets paid on (a) the first call with a new prefix, (b) the first call after TTL expiry, or (c) the first call after a prefix-changing squash.

Figure 1. The "pay twice" pattern. A TTL expiry at call C costs cache_write. A few calls later, hitting 275k forces a squash, and call E pays cache_write again on the new (truncated) prefix.

The insight: if the cache just died (or is about to), and we're already in the truncation range, do the squash now so we only pay cache_write once.

CAP (Cache Anchor Point) helps suppress TTL expiries via periodic refresh pings. The range gate fires only when CAP fails to keep the cache alive — which is automatic in the implementation, because successful CAP pings already advance state.LastLLMCallTime, so the elapsed-time signal collapses naturally.

4. The decision rule

The decision is made before the LLM call is dispatched. Once the call is sent, any cache invalidation has already been paid.

4.1 The TTL anchor

The cache-liveness signal is a single time.Duration computed in the elite layer:

elapsed = now − state.LastLLMCallTime
cache_likely_invalid = (elapsed > 0) AND (elapsed ≥ cap.CacheTTL − cache_ttl_guard)

state.LastLLMCallTime is updated by every successful LLM call AND every successful CAP refresh (core/workflows/elite/cap.go:85). So when CAP is running, its pings naturally push the anchor forward — the range gate doesn't need to filter pings explicitly. When CAP is off, the anchor is just the user's last call.

The elapsed = 0 case (first iteration, or anchor unset) is treated as "cache alive" — no spurious fires on the first iteration after workflow start.

4.2 The decision flow

Figure 3. decideSquashReason as a state diagram. The function returns a reason string when squash should fire, or "" otherwise. Reason values: range_threshold_cache_invalidation, range_threshold_hard_floor, legacy_threshold (only on the legacy path), or image_overflow (precedence-after).

4.3 Pure function signature

// core/xcontext/squash.go

func decideSquashReason(config SquashConfig, state SquashState) string {
    if config.ContextWindow <= 0 { return "" }
    tokens := contextTokens(state.Messages, config.SystemPrompt, state.LastUsage)

    if config.RangeThreshold != nil {
        if o := rangeThresholdOutcome(config.RangeThreshold,
            config.ContextWindow, tokens, state.Elapsed); o != rangeThresholdNo {
            return o.String()
        }
    } else if config.CompactionThreshold > 0 {
        threshold := int(float64(config.ContextWindow) * config.CompactionThreshold)
        if tokens >= threshold { return ReasonLegacyThreshold }
    }
    if imageOverflow(config.Policy, state.Messages) { return ReasonImageOverflow }
    return ""
}

// core/xcontext/tiered.go

func rangeThresholdOutcome(p *RangeThresholdPolicy, ctxWindow, tokens int, elapsed time.Duration) rangeThresholdReason {
    if p == nil || ctxWindow <= 0 { return rangeThresholdNo }
    frac := float64(tokens) / float64(ctxWindow)
    switch {
    case frac >= p.Upper:
        return rangeThresholdHardFloor
    case frac >= p.Lower && elapsed > 0 && elapsed >= p.CacheDeadline():
        return rangeThresholdCacheInvalidation
    default:
        return rangeThresholdNo
    }
}

The same decideSquashReason is called twice per iteration: once in the workflow (pre-flight gate) and once in the activity (inside applyBulkCheckpointStrategy). Both calls see the same inputs because squashEvalTime := exec.Now() is captured once and threaded through the activity request as req.Elapsed. Replay-deterministic by construction.

5. Architecture

Two packages touched. No new packages, no new Temporal activities.

Existing fieldNon-opted-in agentsOpted-in agents

context.overflow_threshold Unchanged. Still drives ask_human. Operator must bump the value above range_threshold.upper. Context.Validate() enforces this at agent-load time.
context.auto_compact[] Unchanged — gated by feature_request.auto_compact matching strategy name. Unchanged. Tier 2 (summarisation) is a separate follow-up PR; this design covers truncation only.
preserve_last_n, truncation_length, squashing_strategy Unchanged. Unchanged. The range gate only changes when truncation fires, not what content survives. bulk_checkpoint is reused byte-for-byte.

Implementation: in core/xcontext/squash.go decideSquashReason: 

if config.RangeThreshold != nil {
    // range gate decides — legacy bypassed
    if o := rangeThresholdOutcome(...); o != rangeThresholdNo { return o.String() }
} else if config.CompactionThreshold > 0 {
    // legacy path
    if tokens >= threshold { return ReasonLegacyThreshold }
}

The mutually-exclusive if/else if guarantees the legacy threshold is never AND-ed with the range gate.

8.1 Configuration schema

context:
  squashing_strategy: bulk_checkpoint
  threshold: 0.275            # retained as fallback (inert when range_threshold is set)
  preserve_last_n: 10
  truncation_length: 150
  overflow_threshold: 0.34    # must exceed range_threshold.upper

  range_threshold:
    lower: 0.200              # 200k on 1M — cache-aware fire begins
    upper: 0.325              # 325k — hard squash floor
    cache_ttl_guard: 15s      # fire when elapsed ≥ (5m − 15s) = 4m45s

8.2 Go schema

type Context struct {
    // ... existing fields, untouched ...
    RangeThreshold *RangeThresholdConfig `json:"range_threshold,omitempty" yaml:"range_threshold,omitempty"`
}

type RangeThresholdConfig struct {
    Lower         float64       `json:"lower" yaml:"lower"`
    Upper         float64       `json:"upper" yaml:"upper"`
    CacheTTLGuard time.Duration `json:"cache_ttl_guard" yaml:"cache_ttl_guard"`
}

Why fractions instead of tokens. Matches the existing threshold: 0.8 convention in cortex (which is also a fraction of context_window). Trade-off: if an agent's context_window changes, the same fraction means a different token count. Documented as a footgun; PRD says context-window changes are out of scope.

9. Failure modes

PRD #16 graceful fallback: "never block the call, never spuriously truncate."

ConditionBehaviour
Agent not opted in (RangeThreshold == nil)Legacy CompactionThreshold gate runs unchanged.
First iteration / no anchor (elapsed == 0)cache_likely_invalid = false by construction. Hard floor at upper still fires unconditionally.
CAP disabled / not provisionedNo effect — anchor is just state.LastLLMCallTime updated by user calls.
CAP refresh fails partwayThat refresh doesn't advance the anchor. Subsequent iterations see the elapsed time growing past the cache deadline → range gate fires → preemptive truncate.
cache_ttl_guard >= CacheTTLSetup logs a warning; the cache deadline becomes non-positive; elapsed > 0 && elapsed >= deadline is always true once the first call happens → fires every iteration in range. Caught at setup; should also be caught at Validate.
Malformed config (e.g., lower > upper)Agent fails to load via RangeThresholdConfig.Validate() before any workflow starts.
overflow_threshold <= range_threshold.upperAgent fails to load via Context.Validate() cross-field check.

11. Open risks

  1. Resume-after-pause anchor. If state.LastLLMCallTime is not persisted across worker restarts, a resumed workflow starts with elapsed = 0 and never fires the cache-invalidation reason on the first iteration after resume. The hard floor still fires correctly. Verify whether LastLLMCallTime is part of the persisted state in core/sessions/....
  2. Double evaluation of decideSquashReason. The pre-flight gate (workflow side) and applyBulkCheckpointStrategy (activity side) both call decideSquashReason on the same snapshot. Cheap but redundant. Optional optimisation: pass the reason from gate → activity in the request.
  3. Approximation in cache_creation_avoided_tokens. The metric uses CacheCreationInputTokens + CacheReadInputTokens from the prior call as a proxy for "what we would have paid as cache_creation on the next call without the range gate." Reasonable approximation but not measured. The metric's description should note this.
  4. Reason label cardinality. When status="not_applied", the reason label is empty (""). Creates an awkward series in Prometheus. Either omit when empty or set to "none".
  5. Fraction vs token configuration. All bounds are fractions of context_window. If an agent's context_window ever changes, the same YAML means a different token threshold. PRD's recommended values (200k, 325k) correspond to (0.200, 0.325) on a 1M context window; on a 2M window they'd be (0.100, 0.1625).

12. Follow-ups

  1. Summarisation tier (PRD Tier 2). Adds range_threshold at the auto_compact[] level, reusing the same RangeThresholdConfig. Context.Validate() enforces non-overlap with the squash range. Will land as a stacked PR.
  2. Anchor persistence verification. Confirm whether state.LastLLMCallTime survives a workflow resume. If not, add a session-level persistence column.
  3. Direct cache_creation measurement. Replace the approximation in cache_creation_avoided_tokens with a real before/after measurement using the next LLM call's actual CacheCreationInputTokens.
  4. Tighter cache_ttl_guard validation. Add an upper bound (e.g., guard < CacheTTL / 2) to catch misconfigurations earlier than the setup-time warning.
  5. Direct rangeThresholdOutcome tests. Current tiered_test.go covers CacheDeadline only; the decision branches deserve dedicated table tests with boundary cases.
