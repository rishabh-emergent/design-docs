Range-Based Truncation for Opus 4.7 (275k Farm Agents)

Date: 2026-05-22  ·  Status: Draft design (v0.2, post-critique)  ·  Scope: Opus 4.7 farm agents on 1M context window  ·  Hard requirement: backward compatible with existing context config; works whether CAP is enabled or not

1. TL;DR

Today, Opus 4.7 farm agents squash exactly when prompt_tokens crosses 275k. Anthropic's prompt cache is prefix-based with a 5-minute TTL. A TTL-driven invalidation that lands shortly before 275k forces a cache-write on the post-invalidation call, then a few calls later the 275k squash fires and we pay a cache-write again on the new (truncated) prefix. We pay for cache invalidation twice.

This design adds a cache-TTL-aware trigger on top of the existing squash mechanism. Two configurable ranges sit below and above the hard threshold:

  • Tier 1 — truncation range [200k, 325k]. If the cache is predicted to be invalidated on the next call (based on wall-clock elapsed since the last call), truncate now using the existing bulk_checkpoint strategy. Hard floor at 325k: if we reach it without invalidation, truncate anyway.
  • Tier 2 — summarisation range [325k, 350k]. Same rule, but invoke summarisation through the existing auto_compact pipeline. Hard ceiling at 350k.

Two hard contracts (v0.2)

  • Works without CAP. The wall-clock prediction uses state.LastLLMCallTime as the primary anchor. If CAP is also running, its successful pings push the anchor forward — but the feature does not require CAP to be enabled. Opting in does not depend on CAP.
  • Fully backward compatible. Existing threshold, overflow_threshold, and auto_compact fields keep their existing semantics. Agents that don't opt in see zero behaviour change. Opt-in is per-agent via a single new YAML block.

What changed in v0.2 vs v0.1

v0.1 placed prediction in cap.Service.PredictNextCallInvalidation. Architect critique: that couples the feature to CAP being deployed and provisioned, which violates the "works without CAP" contract. v0.2 folds prediction into the pure DecideRangeAction function in core/xcontext/ and treats CAP pings as an optional input. Two new files removed; one new function dropped. See §11 for the full critique walkthrough.

2. The problem we're solving

Anthropic's prompt cache is prefix-based with a 5-minute TTL. When the prefix is in the cache, a call pays cache_read (cheap). When it's not, a call pays cache_creation (12.5× the read rate). The expensive cache-creation happens on (a) the first call with a new prefix, or (b) the first call after TTL expiry, or (c) the first call after a prefix-changing squash.

For an Opus 4.7 farm agent running with threshold: 0.275 on a 1M context window, here is the failure case the PRD targets:

Figure 1. The "pay twice" pattern. A TTL expiry at call C costs cache_write. A few calls later, hitting 275k forces a squash, and call E pays cache_write again on the new (truncated) prefix.

The insight: if we know cache-creation just happened (or is about to), and we're already in a range where a squash is likely soon, do the squash now and absorb only one cache-write instead of two.

CAP (Cache Anchor Point) helps suppress TTL expiries via periodic refresh pings — but pings can fail, and the user can simply pause for longer than CAP's refresh budget. This rule must therefore work whether CAP is on or off. When CAP is on, its successful pings refine the prediction (we know the cache was alive at the ping time). When CAP is off, we rely on state.LastLLMCallTime alone.

4. The decision rule

The decision is made before the LLM call is dispatched, in the iteration loop. Once the call is sent, any cache invalidation has already been paid for. It is a single pure function.

4.1 The TTL anchor

The wall-clock prediction uses an anchor — the most recent point in time at which we have evidence the cache was alive:

anchor = max(state.LastLLMCallTime, lastSuccessfulCAPPing)
cache_likely_invalid = (now − anchor) ≥ 5 minutes

state.LastLLMCallTime is the primary, always-available signal — every workflow updates it after each LLM call. lastSuccessfulCAPPing is the most recent CAP ping with source = post_exit AND cache_hit = true — but it's optional. If CAP is disabled, not provisioned, or its ping list is empty, the value is the zero time.Time and max(...) reduces to state.LastLLMCallTime.

Environmentanchor reduces toResult
CAP off / not provisionedstate.LastLLMCallTimePrediction works using just the user's call cadence.
CAP on, no successful pings yetstate.LastLLMCallTimeSame as above. Pings haven't yet refined the signal.
CAP on, recent successful pingmax(LastLLMCallTime, pingTime)If a ping fired between the user's last call and now, the cache was alive at pingTime; we won't spuriously fire.
CAP on, ping returned cache_hit=falsestate.LastLLMCallTimeFailed pings don't advance the anchor; this is correct because they didn't keep the cache alive.

Figure 3. DecideRangeAction as a state diagram. Two early exits (spec nil; anchor zero) protect the failsafe: opted-out or first-iteration cases never engage the new rule and fall through to legacy.

4.3 Why hard floors are enum values, not overflow_threshold

Cortex already has a hard stop driven by overflow_threshold in core/workflows/elite/helpers.go via checkContextOverflow. It fires ask_human — i.e., it stops the workflow. That is the wrong behaviour for the Tier 1 hard floor, which must truncate and continue. So:

  • Tier 1 / Tier 2 hard floors are explicit enum values in DecideRangeAction (HardTruncate / HardSummarise). They fire above the configured upper bound of each range and dispatch the same action as the cache-invalid branch.
  • overflow_threshold is bumped from 0.3 to 0.4 in opted-in YAMLs only to make room. It still fires ask_human as before, just at 400k instead of 300k.

5. Architecture

Two packages collaborate. core/cap/ is consulted opportunistically but is not a dependency.

Figure 4. Two-package architecture. core/cap/ is a dashed box: if it's deployed and provisioned, its setup.Cap.Pings sharpens the prediction. If not, the prediction degrades cleanly to using state.LastLLMCallTime alone. maybeApplyAutoCompact gains one new parameter to support Tier 2's forced invocation; no new helper, no extraction.

5.1 Single pure function

// core/xcontext/rangebased.go

package xcontext

import (
    "time"
    "github.com/emergentbase/cortex/core/cap"      // for CacheTTL constant
    "github.com/emergentbase/cortex/pkg/agentsdk"
)

type RangeAction int

const (
    RangeActionNone RangeAction = iota
    RangeActionTier1Truncate
    RangeActionHardTruncate
    RangeActionTier2Summarise
    RangeActionHardSummarise
)

type RangeDecisionInput struct {
    Spec               *agentsdk.RangeBasedTruncation
    Tokens             int
    LastLLMCallTime    time.Time
    LastSuccessfulPing time.Time   // zero if CAP unavailable
    Now                time.Time
}

type RangeDecisionResult struct {
    Action RangeAction
    Reason string  // "disabled" | "no_anchor_yet" | "below_range"
                   // | "tier1_cache_warm" | "tier1_cache_expiring" | "tier1_hard_floor"
                   // | "tier2_cache_warm" | "tier2_cache_expiring" | "tier2_hard_ceiling"
}

// Pure. No I/O. Safe to call from workflow code.
// Works whether CAP is enabled or not — LastSuccessfulPing zero is a valid input.
func DecideRangeAction(in RangeDecisionInput) RangeDecisionResult {
    if in.Spec == nil || !in.Spec.Enabled {
        return RangeDecisionResult{RangeActionNone, "disabled"}
    }

    anchor := in.LastLLMCallTime
    if in.LastSuccessfulPing.After(anchor) {
        anchor = in.LastSuccessfulPing
    }
    if anchor.IsZero() {
        return RangeDecisionResult{RangeActionNone, "no_anchor_yet"}
    }
    cacheLikelyInvalid := in.Now.Sub(anchor) >= cap.CacheTTL

    // Bands checked highest-to-lowest so hard ceilings/floors win.
    if in.Tokens >= in.Spec.Tier2.UpperTokens {
        return RangeDecisionResult{RangeActionHardSummarise, "tier2_hard_ceiling"}
    }
    if in.Tokens >= in.Spec.Tier2.LowerTokens {
        if cacheLikelyInvalid {
            return RangeDecisionResult{RangeActionTier2Summarise, "tier2_cache_expiring"}
        }
        return RangeDecisionResult{RangeActionNone, "tier2_cache_warm"}
    }
    if in.Tokens >= in.Spec.Tier1.UpperTokens {
        return RangeDecisionResult{RangeActionHardTruncate, "tier1_hard_floor"}
    }
    if in.Tokens >= in.Spec.Tier1.LowerTokens {
        if cacheLikelyInvalid {
            return RangeDecisionResult{RangeActionTier1Truncate, "tier1_cache_expiring"}
        }
        return RangeDecisionResult{RangeActionNone, "tier1_cache_warm"}
    }
    return RangeDecisionResult{RangeActionNone, "below_range"}
}

// LastSuccessfulPostExitPing extracts the most recent successful CAP ping
// from setup.Cap.Pings. Returns zero time if CAP is unavailable or the
// list contains no successful post_exit pings.
//
// This helper is a tiny convenience — callers may inline it.
func LastSuccessfulPostExitPing(pings []cap.Ping) time.Time {
    var t time.Time
    for _, p := range pings {
        if p.Source == "post_exit" && p.CacheHit && p.At.After(t) {
            t = p.At
        }
    }
    return t
}

That's the entire new file. ~80 lines including comments and the helper.

7. Backward compatibility

This is a hard contract. Agents that don't opt in must see zero behaviour change.

Existing config fieldUsed byBehaviour for non-opted-in agentsBehaviour for opted-in agents
context.threshold NeedsSquashing → 275k legacy squash Unchanged. Still triggers bulk_checkpoint at threshold × context_window. Unused at runtime — the range-based decision pre-empts the legacy threshold check (see dispatch in §8.3). Stays in YAML as a documented fallback.
context.overflow_threshold checkContextOverflow → ask_human Unchanged. Still drives ask_human. Operator bumps its value (e.g., 0.3 → 0.4 on 1M ctx) when adding the range-based block, so ask_human fires only above the Tier 2 ceiling.
context.auto_compact[] maybeApplyAutoCompact via feature_request.auto_compact match Unchanged. Gated by feature_request.auto_compact exactly as today. Unchanged for legacy invocations. Tier 2 calls maybeApplyAutoCompact with a force argument that bypasses the gate but uses an independent synthesised config — no interaction with existing auto_compact[] entries.
context.preserve_last_n, context.truncation_length bulk_checkpoint strategy Unchanged. Unchanged. Range-based Tier 1 reuses bulk_checkpoint byte-for-byte, so these fields keep their existing role.
context.squashing_strategy Strategy selection in ApplySquash Unchanged. Unchanged. Must be bulk_checkpoint for opt-in (validated at agent load).

The dispatch switch in flow.go selects the legacy path whenever RangeBasedTruncation == nil or RangeActionNone is returned. The legacy code path is reached by fall-through, not by a separate code branch — same lines as today, just guarded by an outer switch. See §8.3.

9. Failure modes

PRD #16: "If TTL state cannot be determined for any reason, fall back to today's behavior — never block the call, never spuriously truncate or summarise."

The failsafe is data-driven, not exception-driven. Every degenerate input produces RangeActionNone, which falls through to the legacy needsSquashing path:

FailureWhereBehaviour
Agent not opted in (Spec == nil or !Enabled)DecideRangeActionNone / "disabled". Legacy path runs.
CAP off / not provisioned / setup.Cap == nilflow.go pre-calllastPing stays zero. Decision uses just LastLLMCallTime. Feature still works.
CAP on but BigTable ping read failedsetup.Cap.Pings emptySame as "CAP off" — degrades to LastLLMCallTime only.
First iteration, no LastUsage and no pingsDecideRangeActionanchor is zero → None / "no_anchor_yet". Legacy path runs (tokens are below range anyway).
Malformed YAML (e.g., tier1.upper < tier1.lower)YAML unmarshal / agent loadParse error fails agent load before any workflow starts. (For v0 we don't add a custom Validate — Go's strict YAML behaviour on bad shapes plus sensible defaults at decision time is enough.)

None of these paths introduce a new error class, raise an alert, or block the workflow. The Reason tag surfaces in the elite.rangebased.decision_total{reason} metric, so an alarm on unexpected reason rates (e.g., "no_anchor_yet" on a mid-session workflow) detects regressions.

11. Architect critique applied

v0.1 of this design was synthesised from three architects' debate. After re-reading with the user's two hard requirements (CAP-independence; backward compatibility) and an architect-style critique pass, the design shrank by roughly half. The simplifications:

v0.1 (rejected)v0.2 (chosen)Reason
Prediction lived in core/cap/prediction.go as Service.PredictNextCallInvalidation, returning a PredictionResult struct. Prediction is folded into DecideRangeAction as wall-clock math on (LastLLMCallTime, LastSuccessfulPing, now). No new file, no new method, no PredictionResult type. Critical: prediction must work when CAP is off. Putting it on cap.Service couples the feature to CAP being deployed. Folding it into the pure decision function removes the coupling.
Tier 2 dispatch required extracting runAutoCompactPipeline from maybeApplyAutoCompact (~30 LoC refactor). maybeApplyAutoCompact gains one force *AutoCompactConfig parameter. ~10 LoC. Existing callers pass nil. The extraction risked behaviour drift between two callers of the same pipeline. A parameter is smaller and keeps a single code path.
4 metrics: decision_total, tokens_at_decision, cache_creation_avoided_tokens, summariser_cost_usd. 2 metrics: decision_total, tokens_at_decision. Avoidance-token estimation requires post-LLM-call plumbing (delta between predicted and actual cache_creation). Summariser cost requires usage plumbing from activityGenerateSummary. Both useful, neither blocks v0. Defer.
RangeBasedTruncation.Validate() method (~20 LoC). No Validate. YAML parse and runtime defaults handle malformed configs. Validation can be added when it earns its keep. For v0, "tier1.upper < tier1.lower" simply produces None for tokens above tier1.upper because the band check fails — a noisy but safe outcome that surfaces via the reason label.
Three concrete files in core/cap/ and core/xcontext/ + 5 places to touch. One new file (core/xcontext/rangebased.go) + 4 places to touch. Smaller surface, smaller import graph, smaller PR.

Critically, none of the requirements were dropped — Tier 1/2 fires, hard floors, CAP-aware prediction, opt-in, backward compat, multi-truncation, graceful fallback all remain. The PRD's behavioural surface is unchanged. Only the implementation surface shrank.

11.1 What we still could drop but chose to keep

  • The Reason tag in RangeDecisionResult. It's not strictly needed for behaviour. But the cost of debugging a misfire without it is enormous, so it stays.
  • The 5-value enum (Tier1Truncate, HardTruncate, Tier2Summarise, HardSummarise, None). Could collapse to 3 (None, Truncate, Summarise) — hard floors and range fires take the same code path. But the dispatch site benefits from knowing which fire it was, for metric labels and future divergent behaviour. The 5-value enum costs nothing.
  • The exec.GetVersion replay gate on the new dispatch. Strictly speaking, only opted-in agents can take the new path, so non-opted-in replays are safe by construction. But the gate is one line and matches the existing cortex pattern — keep it.

12. Open questions

  1. Tier 1 minimum gap (anti-thrashing). Multi-truncation works by design (PRD #5), but if tokens hover at 200k–210k with the cache repeatedly expiring, we could fire Tier 1 on every other call. Worth adding an optional min_gap_seconds (e.g., 60s) between Tier 1 fires? Or do we trust that bulk_checkpoint's cutoff-advancement-only-forward naturally limits it?
  2. Tier 2 prompt config — duplicate or share? The new tier2.summary_prompt_name is conceptually adjacent to the existing auto_compact[].summary_prompt_name. We chose duplication (independent value) to keep the new block self-contained and avoid coupling. Acceptable, or should Tier 2 reference an existing auto_compact[] entry by name?
  3. Avoidance-token measurement plan. For post-launch sizing of PRD success criteria, we need to estimate cache_creation tokens we saved. The simplest method is to compare the pre-truncation token count to the post-LLM-call cache_creation_tokens. Is that acceptable, or do we need a more principled measurement?
  4. Rollout staging. PRD says rollout is out of scope. But shipping all 7 farm agents simultaneously is risky. Roll out one (e.g., portfolio_expert_agent_opus_4_7) first for ~48h, watch the metric panel, then proceed?
