1. TL;DR

PR #348 adds two new MCP tools to server-tools, both backing the same integration_playbook_expert_v2 contract that Perplexity serves today:

Tool nameBackendMeasured (5 real prod inputs)
integration_playbook_expert_v2_exaExa Agent API — POST /agent/runs, effort=medium, async pollingavg 78.5 s · flat $0.10/call · ~11k char playbooks · 7–16 real doc-page citations per call
integration_playbook_expert_v2_geminiGemini 3.1-pro-preview + google_search groundingavg 32.9 s · ~$0.049/call · ~10k chars · grounding URLs rarely surfaced

Baseline for comparison — Perplexity sonar-deep-research on the same inputs: avg 277.7 s, ~$1.00/call, ~73k chars. Both new tools use the same EA verified-playbook lookup as the Perplexity tool — only the fallback (when no verified playbook exists in DB) calls a different provider.

Why three twins instead of one replaceable backend? Cortex flips which tool name an agent yaml whitelists per-cohort via Unleash, without redeploying server-tools. All three tools alias to the same display_name in the agent yaml's overrides.tools, so the LLM-visible surface is identical regardless of provider.

4. pkg/tools/exa/client.go — Exa Agent API client

4.1 Constants

const (
    exaBaseURL    = "https://api.exa.ai"
    agentRunsPath = "/agent/runs"

    // ExaBetaHeader gates the Agent API (per Exa's integration guidance).
    ExaBetaHeader = "agent-2026-05-07"

    // DefaultEffort caps cost at $0.10/call ("medium" tier per Exa Agent pricing).
    DefaultEffort = "medium"

    // pollInterval follows Exa's documented recommendation of 4s.
    pollInterval = 4 * time.Second
    maxWait      = 10 * time.Minute

    // maxConsecutivePollFailures tolerates transient blips without abandoning an in-flight (billed) run.
    maxConsecutivePollFailures = 3
)
The system/user prompt templates used to live here as a verbatim Perplexity copy; they now come from the shared pkg/tools/playbook package (playbook.BuildPrompts) — one source of truth for all three provider twins.
Why effort=medium? Exa Agent's effort tiers price at low $25/1k, medium $100/1k, high $500/1k. Medium = flat $0.10/call max — measured exactly $0.10 on all 5 replayed prod inputs, vs Perplexity's $0.84–$1.26 float. Predictable billing was an explicit selection criterion.

4.3 Execute() — the public entrypoint

Signature mirrors perplexity.Client.Execute 1:1 so the tool layer stays provider-agnostic.

func (c *Client) Execute(ctx context.Context, query string, techStack []string,
                         jobID, userID, envImage string) (Response, error) {
    if c.apiKey == ""      { return Response{}, fmt.Errorf("EXA_API_KEY ...") }
    if len(techStack) == 0 { techStack = []string{"FastAPI", "React"} }

    stackStr     := strings.Join(techStack, ", ")
    systemPrompt := fmt.Sprintf(systemPromptTemplate, stackStr)
    userPrompt   := fmt.Sprintf("Create a detailed integration playbook for %s in %s ...", query, stackStr)
    agentQuery   := systemPrompt + "\n\n" + userPrompt   // single query field for the Agent API

    run, err := c.startRun(ctx, agentQuery)
    if err != nil { return Response{}, err }
    if !isTerminal(run.Status) {
        run, err = c.poll(ctx, run.ID)                   // 4s interval, max 10 min
        if err != nil { return Response{}, err }
    }
    if run.Status != "completed" {
        return Response{}, fmt.Errorf("exa agent run ended with status=%s stop_reason=%s ...", ...)
    }
    if run.Output == nil || run.Output.Text == "" {
        return Response{}, fmt.Errorf("exa agent run completed with empty output ...")
    }
    return toResponse(*run), nil
}

Line-by-line — easy English

StepWhat it does
key / stack guardsFail fast without a key; default to FastAPI+React if EA didn't supply a stack.
agentQueryThe Agent API takes ONE query string, not role-based messages — so the same system + user templates used by the Perplexity twin are concatenated with a blank line.
startRunPOST /agent/runs. Returns the run object immediately (usually status=queued).
isTerminal checkIf Exa already returned a terminal status on create (possible for failures), skip polling.
pollGET /agent/runs/{id} every 4 s until terminal. Measured: 50–113 s for medium effort.
status / empty guardsfailed/cancelled runs and completed-but-empty outputs become explicit errors instead of silently handing the agent nothing.
toResponseWrap output.text into the Perplexity-shaped Response; dedupe grounding URLs into Citations (first-seen order).

4.4 doRunRequest() — one HTTP path for create + poll

// doRunRequest sends one Agent API request and decodes the run object.
func (c *Client) doRunRequest(ctx context.Context, method, url string, body io.Reader) (*agentRunResponse, error) {
    req, _ := http.NewRequestWithContext(ctx, method, url, body)
    req.Header.Set("x-api-key", c.apiKey)
    req.Header.Set("Exa-Beta", ExaBetaHeader)
    req.Header.Set("Content-Type", "application/json")

    resp, err := utils.RunWithNewRelic(ctx, req, c.client)   // shared pool + NR instrumentation
    ...
}

startRun (POST with body) and the poll loop (GET) both delegate here — one place for headers, status checks, and decoding.

4.5 Poll-failure tolerance

out, err := c.doRunRequest(ctx, http.MethodGet, exaBaseURL+agentRunsPath+"/"+runID, nil)
if err != nil {
    // Transient poll failures must not abandon an in-flight (billed) run.
    consecutiveFailures++
    if consecutiveFailures >= maxConsecutivePollFailures {
        return nil, fmt.Errorf("exa poll failed %d times in a row ...", ...)
    }
    continue
}
consecutiveFailures = 0
An Exa run is billed once it starts. A single network blip during the 1–2 minute polling window previously would have abandoned the run and wasted the $0.10. Now up to 3 consecutive poll failures are tolerated; the counter resets on any success.

4.6 Unit tests (client_test.go)

  • TestIsTerminal — the full status table (queued/running false, completed/failed/cancelled true).
  • TestToResponse_DedupesCitationsFirstSeenOrder — duplicate URLs collapse, empties skipped, order preserved.
  • TestToResponse_NilOutput — nil output never panics.

5. pkg/tools/exa/tool.go — MCP wrapper

Behaviorally identical to pkg/tools/perplexity/tool.go, but the shared parts now live in pkg/tools/playbook — the tool file keeps only the provider-specific call and its events:

MethodWhat it does
Executeplaybook.ExtractQuery → playbook.CheckKnowledgeBase, returns the cached verified playbook on hit, else p.client.Execute() (the Agent API path). Emits events on every branch.
playbook.CheckKnowledgeBasePOSTs EA's /api/v1/knowledge/check. Honors X_emergent_base_url, falls back to agentsService.EAHostFor(ctx) (per-request X-EA-Host override), and passes execCtx.TenantHeadersMap() — identical to Perplexity.
IsEnabledp.client != nil && p.client.apiKey != "" — tool hidden without a key.
DescriptionReturns playbook.ToolDescription — byte-identical to Perplexity's (md5-verified), provider-blind LLM surface.
MCPExecuteplaybook.ToMCPResult: appends citations as a Sources: list, wraps as {"content": ..., "cost": 1.5}.
Why the shared package? The 2,000-char description, 577-char prompt template, KB-lookup logic, and metadata shapes were copy-pasted ×3 — and the copies had already drifted once (the tenant-headers/EAHostFor parity bugs caught in review). pkg/tools/playbook makes drift structurally impossible for the twins. Perplexity (the live prod path) is deliberately left untouched in this PR and migrates in a follow-up once the twins are validated.
The cost: 1.5 is intentional — it is an internal billing unit charged to the job (flat on research fallback, KB price on cache hit), matching the Perplexity twin exactly. It is NOT the provider's dollar cost; that is observable via the ExaAPICallCompleted event.

6. pkg/gemini/research.go — grounding helper

Adds Client.Search() to the existing Gemini client: GenerateContent on gemini-3.1-pro-preview with the google_search built-in tool, maxOutputTokens 8192, system instruction + user content from the same templates. Uses the package's existing backend fan-out (Vertex first, direct API fallback).

  • extractText — concatenates the first candidate's text parts.
  • extractCitations — unique URLs from GroundingMetadata.GroundingChunks[].web.uri, first-seen order. In practice Gemini often surfaces zero URLs even when grounding ran — observed across our replays.
A review bot flagged gemini-3.1-pro-preview as unsupported based on a stale models page. Empirically falsified: 34 successful grounded calls (29-integration eval + 5 prod replays), all HTTP 200 with usage metadata.

7. pkg/tools/gemini_research/tool.go — MCP wrapper

Identical structure to exa/tool.go: same delegation to playbook (query extraction, EA-first short-circuit, description, MCP schema, result wrapping). The fallback calls gemini.Client.Search() with prompts from playbook.BuildPrompts and converts gemini.ResearchResponse into playbook.APIMetadata.

8. pkg/events/{types,service}.go — telemetry

ExaEvent and GeminiResearchEvent have the exact same fields as PerplexityEvent; only the event-name prefix differs. Four emit methods per provider:

MethodEmitted when
*KnowledgeBaseFoundEA returned a verified playbook (cache hit, $0).
*KnowledgeBaseNotFoundClassifier returned OTHER — falling through to the live provider.
*APICallCompletedProvider call succeeded; latency, content length, citations, tokens, full response body.
*APICallFailedProvider call errored; duration + error message.

9. internal/mcpserver/registry.go — wiring

// setupExaClient initializes the Exa Agent API client.
func setupExaClient(httpClient *http.Client) *exa.Client {
    if apiKey := os.Getenv("EXA_API_KEY"); apiKey != "" {
        return exa.NewClient(apiKey, httpClient)
    }
    return nil
}

Single-key gate (the earlier two-step pipeline also needed ANTHROPIC_API_KEY; the Agent API removed that dependency). Tool registration: 

// setupExaTool creates the Exa research tool (integration_playbook_expert_v2_exa).
// setupGeminiResearchTool creates the Gemini research tool (integration_playbook_expert_v2_gemini).
//   → refuses to register unless GEMINI_API_KEY or both Vertex env vars are set, because
//     setupGeminiClient returns non-nil even with zero usable backends.
If a key is missing the tool simply isn't registered — environments without Exa/Gemini config gracefully stay Perplexity-only.

10. End-to-end dry run

Scenario: agent calls integration_playbook_expert_v2_exa with {"query": "INTEGRATION: Stripe payment integration for Expo... CONSTRAINTS: ..."}; no verified playbook in the EA DB; expo template. Numbers are from a real replayed prod input.

  1. Cortex calls the MCP tool. The MCP server routes to ExaTool.MCPExecute with the agent's tool input.
  2. Execute picks the query (the full INTEGRATION/CONSTRAINTS block) and calls checkKnowledgeBase.
  3. EA knowledge check. POST {baseURL}/api/v1/knowledge/check with query, run/user ids, env image, tenant headers. (~200 ms)
  4. EA returns knowledge_found=false + tech_stack=["expo react native","mongodb database","react native web"].
  5. Event: ExaKnowledgeBaseNotFound published.
  6. Agent run starts. POST https://api.exa.ai/agent/runs with {"query": "<system template + user template>", "effort": "medium"} and the Exa-Beta header. Run comes back status=queued.
  7. Polling. GET /agent/runs/{id} every 4 s. Exa's agent searches the web (~9 searches on this input), reads sources, writes the playbook.
  8. Run completes — status=completed, stopReason=schema_satisfied, ~92 s total. costDollars.total = 0.10.
  9. toResponse wraps output.text (~10.6k chars) and dedupes 14 grounding URLs into Citations.
  10. Event: ExaAPICallCompleted with latency, sizes, and the full response body.
  11. MCPExecute appends the Sources: list (14 real doc URLs: docs.stripe.com, docs.expo.dev, ...) and returns {"content": "...", "cost": 1.5} to cortex.
  12. The agent reads the playbook and writes the integration. Net: ~92 s wall-clock, $0.10 provider cost — vs ~258 s and $0.84 for the same input on Perplexity.
