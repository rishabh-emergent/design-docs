1. TL;DR

PR #348 adds two new MCP tools to server-tools, both backing the same integration_playbook_expert_v2 contract that Perplexity serves today:

Tool nameBackendWhy
integration_playbook_expert_v2_exaExa /search (semantic) + Claude Haiku 4.5 synthesis~8× cheaper than Perplexity, similar substance.
integration_playbook_expert_v2_geminiGemini 3.1-pro-preview + google_search grounding~24× cheaper than Perplexity, best substance scores in the 4-way eval.

Both tools use the same EA verified-playbook lookup as the Perplexity tool — only the fallback (when no verified playbook exists in DB) calls a different provider.

Why three twins instead of one replaceable backend? Cortex can flip which tool name a given agent yaml whitelists per-cohort via Unleash, without re-deploying server-tools. All three tools alias to the same display_name in the agent yaml's overrides.tools, so the LLM-visible surface is identical regardless of provider.

4. pkg/tools/exa/client.go — two-step pipeline

4.1 Constants and types

const (
    exaBaseURL       = "https://api.exa.ai"
    anthropicBaseURL = "https://api.anthropic.com"

    // Claude Haiku 4.5 — synthesis model id; keep in sync with config/model_config.go.
    SynthesisModel = "claude-haiku-4-5-20251001"

    DefaultNumResults     = 15
    DefaultMaxOutputChars = 8000

    systemPromptTemplate = `You are Integration_Playbook_Expert, ...` // verbatim from Perplexity
)
Why two base URLs in one client? The pipeline talks to both Exa (/search) and Anthropic (/v1/messages) inside a single Execute() call. Keeping both URLs as package constants makes the intent obvious at the call site without grepping for hardcoded strings. 
type Client struct {
    client       *http.Client
    exaAPIKey    string
    anthropicKey string
}

func NewClient(exaKey, anthropicKey string, httpClient *http.Client) *Client { ... }

Same shape as perplexity.Client but with two API keys. The http.Client is injected by registry.go so all tools share the same connection pool, retry policy, and New Relic instrumentation (utils.RunWithNewRelic).

4.2 The wire structs (Exa /search + Anthropic Messages)

StructUsed for
exaSearchRequestPOST body to https://api.exa.ai/search. Fields: query, numResults, type: "neural", useAutoprompt: true, plus a contents sub-object asking Exa to return text + highlights + summary per result.
exaSearchResult / exaSearchResponseDecoded response: array of {url, title, publishedDate, summary, highlights[], text}.
anthropicRequestPOST body to https://api.anthropic.com/v1/messages. Minimal subset — only model, max_tokens, system, messages[].role/content.
anthropicResponseDecoded response: array of content blocks (we keep only type: "text"), plus usage.input_tokens / output_tokens.

4.3 Execute() — the public entrypoint

This is the only method tool.go calls. Signature matches Perplexity 1:1.

func (c *Client) Execute(ctx context.Context, query string, techStack []string,
                         jobID, userID, envImage string) (Response, error) {
    if c.exaAPIKey == ""    { return Response{}, fmt.Errorf("EXA_API_KEY ...") }
    if c.anthropicKey == "" { return Response{}, fmt.Errorf("ANTHROPIC_API_KEY ...") }
    if len(techStack) == 0  { techStack = []string{"FastAPI", "React"} }

    stackStr := strings.Join(techStack, ", ")
    systemPrompt := fmt.Sprintf(systemPromptTemplate, stackStr)
    userPrompt   := fmt.Sprintf("Create a detailed integration playbook for %s in %s ...", query, stackStr)

    searchQuery := fmt.Sprintf("%s %s integration setup guide official documentation",
                               query, strings.Join(techStack, " "))

    searchResp, err := c.searchExa(ctx, searchQuery)             // step 1
    if err != nil || len(searchResp.Results) == 0 { ... }

    sourceContext, citations := formatSearchResults(searchResp.Results)

    synthesized, usage, err := c.synthesizeWithHaiku(ctx,         // step 2
                                  systemPrompt, userPrompt, sourceContext)
    if err != nil { return Response{}, ... }

    return Response{
        ID:    fmt.Sprintf("exa-search-haiku-%d", time.Now().UnixNano()),
        Model: SynthesisModel,
        Choices: []Choice{{Index: 0, FinishReason: "stop",
                            Message: Message{Role: "assistant", Content: synthesized}}},
        Usage:     Usage{PromptTokens: usage.InputTokens, CompletionTokens: usage.OutputTokens,
                         TotalTokens: usage.InputTokens + usage.OutputTokens},
        Citations: citations,
    }, nil
}

Line-by-line — easy English

LineWhat it does
if c.exaAPIKey == ""If we never got the Exa key from environment, fail fast — don't make a half-baked call.
if c.anthropicKey == ""Same check for the Anthropic key. The pipeline needs BOTH to work.
if len(techStack) == 0If the EA classifier didn't tell us which stack to write code for, assume the most common one (FastAPI + React).
systemPrompt := fmt.Sprintf(...)Bake the stack into the persona prompt so Haiku writes code in the right language/framework.
userPrompt := fmt.Sprintf(...)Build the user-facing ask: "give me a playbook for X in Y stack".
searchQuery := ...Build a separate, documentation-oriented query for Exa. The "integration setup guide official documentation" suffix biases Exa's semantic search toward real docs and away from blog spam.
c.searchExa(ctx, searchQuery)HTTP POST to https://api.exa.ai/search — wait for the JSON response of up to 15 docs.
len(searchResp.Results) == 0Defensive: if Exa returned no hits, don't even try synthesis (Haiku would have no context to work with).
formatSearchResults(...)Turn the array of docs into one big text block ("[Source 1]... URL: ... Content: ..."), and collect the URLs into the citations slice we'll return.
c.synthesizeWithHaiku(...)HTTP POST to Anthropic Messages API. Passes the same Integration_Playbook_Expert system prompt + user prompt + the retrieved docs as context.
The return Response{...} blockStuff Haiku's markdown into a Choice shaped like Perplexity's response so tool.go doesn't know which provider ran.

4.4 searchExa()

Standard HTTP POST. Marshals the request body, sets x-api-key, sends through the shared HTTP client (so New Relic spans are recorded). Non-2xx → returns a wrapped error with the response body included for debugging.

4.5 formatSearchResults()

Pure function (no I/O). Loops over results and builds a single context string. Each source block looks like: 

[Source 3] Stripe Payment Element guide
URL: https://docs.stripe.com/payments/payment-element
Published: 2025-08-12
Summary: A drop-in payment UI for one-time and recurring charges.
Key highlights:
- The Payment Element renders all enabled payment methods automatically.
- ...

Content:
<first 8000 chars of the page>

Blocks are joined with \n\n---\n\n. Returns the joined text and a parallel slice of URLs.

4.6 synthesizeWithHaiku()

Wraps the retrieved sources into a structured user message and calls Claude Haiku 4.5:

userMsg := fmt.Sprintf(`%s

You have access to the following web sources retrieved via search. Use them to
construct an accurate, current playbook. Cite sources inline as [N] (matching
source numbers below).

================================================================================
RETRIEVED SOURCES
================================================================================
%s
================================================================================

Now produce the integration playbook. Be specific (exact package versions, real
endpoint URLs from the sources, exact env var names). Include code that compiles.
End with a SOURCES section listing each [N] URL used.`, userPrompt, sourceContext)

The hard instruction "include code that compiles" is intentional — the rubric in the eval treats the downstream consumer as a pattern-matching extractor, not a forgiving LLM. We want Haiku to over-specify versions and endpoints.

Non-2xx HTTP → wrapped error. Top-level error object in the response → also wrapped error. Empty content → still an error. Only when we have text do we return it along with the usage block.

This is the same EA-first short-circuit as the Perplexity tool. Verified playbooks in the EA DB cost $0 and return in ~1-2 seconds. The expensive provider path only fires on classifier=OTHER (≈14% of FastAPI+React traffic per prior prod analysis).

6. pkg/gemini/research.go — grounding helper

Adds a single new method, Client.Search(), to the existing pkg/gemini client. The implementation uses the same backend-fan-out logic (Vertex first, then direct API) that the rest of the package uses for image generation and file analysis. 

func (c *Client) Search(ctx context.Context, req SearchRequest) (*ResearchResponse, error) {
    model := req.Model
    if model == "" { model = ModelGeminiResearch } // "gemini-3.1-pro-preview"

    contents := []*genai.Content{
        genai.NewContentFromParts([]*genai.Part{genai.NewPartFromText(req.Query)}, genai.RoleUser),
    }

    config := &genai.GenerateContentConfig{
        ResponseMIMEType: "text/plain",
        MaxOutputTokens:  DefaultMaxOutputTokens,        // 8192
        Tools: []*genai.Tool{
            {GoogleSearch: &genai.GoogleSearch{}},       // the magic — turns grounding ON
        },
    }
    if req.SystemPrompt != "" {
        config.SystemInstruction = &genai.Content{
            Parts: []*genai.Part{genai.NewPartFromText(req.SystemPrompt)},
        }
    }

    clients := c.getAvailableClients()
    for _, gc := range clients {                          // try Vertex, fall back to direct API
        resp, err = gc.Models.GenerateContent(ctx, string(model), contents, config)
        if err == nil && len(resp.Candidates) > 0 { break }
    }

    cand := resp.Candidates[0]
    return &ResearchResponse{
        Content:   extractText(cand),
        Citations: extractCitations(cand),  // pulls unique URLs from GroundingMetadata
        Model:     string(model),
        InputTokens / OutputTokens / TotalTokens: from resp.UsageMetadata,
    }, nil
}

6.1 Cleanup applied during review

The first version had a redundant if/else block: both branches produced the same contents value, and the if branch built a systemPrompt local variable that was immediately discarded with _ = systemPrompt. The system instruction is actually applied via config.SystemInstruction a few lines later. The dead code was removed in commit dc5c480.

6.2 Helpers

  • extractText — concatenates all Text parts from the first candidate. Returns "" if the response is empty or malformed.
  • extractCitations — loops over GroundingMetadata.GroundingChunks, pulls each web.uri, dedupes by URL preserving first-seen order. Returns nil when grounding metadata isn't present (which, per our prod-trace investigation, happens in ~60% of Gemini grounded calls — Gemini's grounding works internally but URLs aren't always surfaced).

Adds 8 new methods: 4 per provider, mirroring Perplexity's pattern.

MethodEmitted when

// setupExaClient initializes the Exa client; requires both EXA_API_KEY and
// ANTHROPIC_API_KEY for the two-step pipeline.
func setupExaClient(httpClient *http.Client) *exa.Client {
    exaKey := os.Getenv("EXA_API_KEY")
    anthropicKey := os.Getenv("ANTHROPIC_API_KEY")
    if exaKey == "" || anthropicKey == "" {
        return nil
    }
    return exa.NewClient(exaKey, anthropicKey, httpClient)
}

If either key is missing, the client stays nil and the corresponding tool's IsEnabled() returns false — the tool simply isn't registered. This is how staging environments without an Anthropic key gracefully fall back to Perplexity-only.

// setupExaTool creates the Exa research tool (integration_playbook_expert_v2_exa).
func setupExaTool(ctx context.Context, deps *Deps) interfaces.MCPCapable {
    if deps.Exa == nil { return nil }
    tool := exa.NewExaTool(deps.Exa, deps.AgentsSvc, deps.ProxyAPIKey, deps.EventSvc,
                            "integration_playbook_expert_v2_exa")
    log.Info(ctx, "Registered MCP tool", log.Str("tool", "integration_playbook_expert_v2_exa"))
    return tool
}

// setupGeminiResearchTool creates the Gemini research tool (integration_playbook_expert_v2_gemini).
func setupGeminiResearchTool(ctx context.Context, deps *Deps) interfaces.MCPCapable {
    if deps.Gemini == nil { return nil }
    tool := gemini_research.NewGeminiResearchTool(deps.Gemini, deps.AgentsSvc,
              deps.ProxyAPIKey, deps.EventSvc, "integration_playbook_expert_v2_gemini")
    log.Info(ctx, "Registered MCP tool",
              log.Str("tool", "integration_playbook_expert_v2_gemini"))
    return tool
}

10. End-to-end dry run

Scenario: agent calls integration_playbook_expert_v2_exa with {"query": "Supabase Auth"}; the integration is not in the EA verified playbook DB (classifier returns OTHER); tech stack is FastAPI + React; we're in dev where both EXA_API_KEY and ANTHROPIC_API_KEY are set.

  1. Cortex calls MCP tool over HTTP. The MCP server routes to ExaTool.MCPExecute with args = {"query": "Supabase Auth"}.
  2. MCPExecute wraps args into interfaces.ToolInput and calls p.Execute(ctx, input, execCtx). execCtx carries RunID, UserID, EnvImage, and request headers.
  3. Execute picks searchQuery = "Supabase Auth" (uses query arg because task is empty).
  4. Knowledge-base check fires. checkKnowledgeBase reads X_emergent_base_url from execCtx.RequestHeaders, falls back to p.agentsService.GetEAHost(). Then POSTs to {baseURL}/api/v1/knowledge/check with {"query":"Supabase Auth","run_id":"...","user_id":"...","env_image":"..."}.
  5. EA returns {"knowledge_found": false, "tech_stack": ["FastAPI","React"]} — no verified playbook for this query.
  6. Event fired: p.eventSvc.ExaKnowledgeBaseNotFound(ctx, execCtx, "integration_playbook_expert_v2_exa", "") publishes to the llm-proxy-events PubSub topic.
  7. Live pipeline begins. apiStartTime := time.Now(). Calls p.client.Execute(ctx, "Supabase Auth", ["FastAPI","React"], runID, userID, envImage).
  8. Inside the Client: builds systemPrompt = "...Complete code examples for this tech stack: FastAPI, React..." and searchQuery = "Supabase Auth FastAPI React integration setup guide official documentation".
  9. Exa /search call. POSTs to https://api.exa.ai/search with the search request body. Receives ~15 source URLs back, each with text + highlights + summary. Total response size ~50 KB.
  10. formatSearchResults builds the source context. Joins all docs into one big string like "[Source 1] Supabase Docs\nURL: ...\nContent: ...\n\n---\n\n[Source 2] ...". Citations slice now holds 15 URLs.
  11. Anthropic Messages call. POST to https://api.anthropic.com/v1/messages with model=claude-haiku-4-5-20251001, system=systemPrompt, and user=userPrompt + "\n\nRETRIEVED SOURCES\n...". Haiku reads ~5-15 KB of input tokens, generates a Markdown playbook (~3-5 KB output).
  12. Synthesis returns. synthesizeWithHaiku returns (markdown, usage{InputTokens: 5234, OutputTokens: 3120}, nil).
  13. Client builds the Response: wraps the markdown into a Choice with FinishReason="stop", fills Usage from the Haiku usage block, attaches the 15 citation URLs.
  14. Back in tool.Execute: apiDuration := time.Since(apiStartTime) ≈ 32s (5s search + 27s synthesis). p.eventSvc.ExaAPICallCompleted fires with all the stats.
  15. Tool returns ToolOutput{Content: markdown, Success: true, Metadata: ExaAPIMetadata{Source:"exa_api", Citations:[...], CitationsCount:15, Cached:false}}.
  16. MCPExecute wraps the response into mcp.CallToolResultText. Appends a "Sources:" footer listing the 15 URLs, embeds everything into a JSON {"content":"...","cost":1.5} blob, returns to cortex.
  17. Cortex stores the result as a tool_result message and feeds it back to the agent on the next iteration. The agent reads the playbook and starts writing code.
Net result: One MCP call → 1 EA HTTP call (knowledge check, ~200 ms) + 1 Exa HTTP call (~5 s) + 1 Anthropic HTTP call (~27 s) → ~32 s wall-clock, ~$0.10 in API cost, one cached telemetry event, agent gets a ~5 KB Markdown playbook with citations.
