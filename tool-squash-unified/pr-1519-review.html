2. File-by-file: kis file me kya change hua

📄 pkg/agentsdk/context.go +127 — config ka YAML surface

Simple terms: Ye file batati hai ki agent YAML me tool_squash: section kaisa dikhega. Naye types: ToolSquashConfig (rules ki list + preserve_last_n_assistant + default truncation_length), ToolSquashRuleConfig (kaunsa tool match kare — tool_name_regex, optionally args_content_regex), aur do actions: OnArgs (sirf truncate) / OnResult (replace ya truncate).

Validation: regex compile hote hain, har rule me kam-se-kam ek action hona chahiye, negative numbers reject. Plus ek cross-invariant: tool_squash.preserve_last_n_assistant ≥ context.preserve_last_n — taaki tool-squash kabhi us "live tail" ko na chhue jo bulk-checkpoint protect karta hai.

📄 core/xcontext/policy.go +54 — YAML → runtime translation

Simple terms: CompileToolSquash() YAML config ko xllm ke runtime ToolSquashPolicy me convert karta hai. Pure mapping code hai, koi logic nahi. Nil/empty config → nil policy (feature off). Action-less rules silently drop hote hain (defense — experiment patch se aayi bad rule workflow crash na kare).

📄 core/workflows/elite/setup.go +38 — workflow me attach karna

📄 core/xcontext/squash.go +75/−7 — sabse important file: cache protection + persistence fix

Do kaam:

(a) StartIndex — cache protection. Anthropic prompt-cache me purane messages cached hote hain. Agar tool-squash unhe mutate kare to cache invalid ho jata hai (paisa barbaad). toolSquashStartForReason() decide karta hai squashing kahan se shuru ho: normally previous cutoff ke baad se (cached prefix safe); agar reason hi cache-invalidation hai to index 0 se (cache waise hi mar chuka hai). withToolSquashStartIndex() policy ki deep-copy banata hai taaki shared policy mutate na ho.

(b) messagesEqual — persistence fix. Squash ke baad cortex original vs squashed messages diff karta hai, jo change hua wo truncated_content_ref me DB me save hota hai. Pehle tool_use.Input ke changes detect hi nahi hote the → truncation har baar dobara hoti. Ab naya BlockTypeToolUse case + jsonEqual() helper. JSON comparison kyu? Kyunki xllm messages ko JSON-clone karta hai jisse int → float64 ban jata hai; reflect.DeepEqual usse "changed" samajh ke faltu DB writes karta. Smart catch. 👍

3. Review findings

F1 Should fix Metric workflow code se emit ho raha hai (context.Background())

recordToolSquashValidation() ko attachToolSquash() call karta hai, jo setup() = workflow-side code hai. Cortex ka apna CLAUDE.md clearly bolta hai: "metrics... must be done inside activities, not from the workflow function" aur "never use context.Background() in production code".

Practical problem: workflow replay (worker restart / sticky-cache eviction) par setup() dobara chalega aur counter dobara increment hoga — "alertable" metric noisy ho jayega.

Sibling precedent: isi function me 30 lines niche, range_threshold misconfig sirf log.Error karta hai aur policy drop karta hai — koi metric nahi. Wahi pattern follow karo.

Suggested fix: metric hatao, warn-log par alert lagao (simplest, sibling-consistent). Ya validation ko ResolveAgentConfig activity me move karo jahan real context hai. Comment me "#1499 ka same pattern" likha hai — par #1499 kabhi merge nahi hua, wo precedent nahi hai.

F2 Merge blocker (known) xllm pseudo-version pin

go.mod me v1.0.88-0.20260610095955-35e10ec264e3 — branch build. xllm PR #200 land + tag hone ke baad re-pin karna hai. PR body ye acknowledge karti hai (par body me purana pseudo-version v1.0.79-... likha hai — wo bhi stale hai).

F3 PR body stale Description code se match nahi karti

Body bolti hai: "applyBulkCheckpointStrategy uses xllm.HasToolSquashWork to fire squash when bulk has no cutoff". Code exactly opposite karta hai — bulk cutoff nahi to tool-squash fire hi nahi hota, aur test TestApplySquash_DoesNotFireWhenBulkCannotTruncate isi behavior ko pin karta hai ("Simpler model, easier cache reasoning" — jo sahi decision hai). xllm HasToolSquashWork export karta hai par cortex use kahin call nahi karta. Body update karo; agar koi consumer nahi to xllm se export hatane ka issue file karo.

F4 Edge case simple strategy ke saath cache protection nahi hai

StartIndex protection sirf bulk_checkpoint path me set hota hai. Agar koi agent squashing_strategy: simple + tool_squash configure kare, to policy unchanged pass-through hoti hai — StartIndex=0, matlab har squash par tool-squash poori history walk karega, cached prefix included. Mutations idempotent hain to damage limited hai, par intent clearly bulk-only hai.

Suggested fix (cheap): attachToolSquash me ek guard — strategy bulk_checkpoint nahi to skip + warn. Ya document karo ki ye bulk-only feature hai.
